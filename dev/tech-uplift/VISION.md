## §1 Premise-check: «Claude Code в чистом виде уже не про мои задачи»

Тезис владельца не принимается на веру и не отвергается — раскладывается по функциям
(правило [[feedback_substrate_premise_verification]]: верифицируй trigger claims до
архитектурных решений). **Критерий оправданности выноса** функции за пределы CC —
хотя бы одно из трёх, доказуемо:

- **(Д) Детерминизм:** промпт/скилл физически не гарантирует исполнение (теорема трека
  semantic-continuity: «знание ≠ исполнение»; калибровочное наблюдение ресерча:
  весь рынок «строг к себе — мягок в поставке» именно потому, что поставляет тон).
- **(М) Межсессионность:** состояние/процесс живёт между сессиями и вне окна контекста
  (статусы, очереди, мониторинг, бюджеты).
- **(Х) Межхарнессность:** функция нужна не только Claude Code-сессии.

| Функция экосистемы | Где живёт сейчас | Критерий | Вердикт |
|---|---|---|---|
| Суждения D1/D2: discovery-диалоги, спеки, качественный анализ | CC: skills/commands | — | **Остаётся в CC.** Это сильнейшая сторона харнесса; ре-хостинг не даёт ничего |
| Исследовательский контур (research-intake, deep-research) | CC: skills + Workflow | — | **Остаётся в CC** |
| Enforcement dev-процесса | Код: process-gate.js + verify-цепь | Д | **Уже вынесен** (образец: так должно быть везде) |
| Валидация артефактов | Код частично: artifact-validate.js (7 из 44 правил) + LLM-раннер | Д | **Довынести**: схемы + расширение автоматики |
| Продуктовый workflow (фазовые переходы D1→handoff) | Промпт-тон команд + hook-валидация артефактов | Д | **Вынести**: FSM guard-переходов кодом — мы сейчас сами «self-strict / shipped-soft» |
| Статус проекта | Размазан по md (ROADMAP/журнал/память) + fabric-engine (только межпроцессный слой) | Д+М | **Вынести**: единый event-store (генерализация fabric-engine) |
| Оркестрация параллельных сессий, очереди, чекпойнты | Вручную + Волна 0 ассист-пульта | М | **Вынести** → Кондуктор (уже решено 0216; здесь — вклад заимствований) |
| Бюджеты токенов / MDP-дисциплина | Документная (CLAUDE.md, договорные потолки) | Д+М | **Вынести**: circuit-breaker кодом (чистая ниша по coverage-check) |
| Качество скиллов | Не проверяется (только каталог-дрейф) | Д | **Вынести**: eval-фреймворк (чистая ниша) |
| Доступ к методологии из других харнессов | Отсутствует (хост-примитивы CC) | Х | **Условно**: MCP-интерфейс дёшево даёт межхарнессность; полный EoS — trigger-gated |

**Вывод premise-check:** тезис подтверждается **частично и точечно**. CC перестаёт
подходить не «вообще», а ровно там, где мы заставляем LLM-слой делать работу рантайма:
гарантии, состояние, счётчики, параллелизм. Ответ — не уход с CC (отвергнут, см. §5),
а **разделение труда**: CC = слой суждений, собственный код-слой = слой гарантий.
Это то же разделение, которое уже формализовано в
[`docs/orchestrator-module/SPEC.md`](../../docs/orchestrator-module/SPEC.md) §2
(«скелет — строгий детерминизм / суждение агента — ограничено / verification-гейт —
строгий») — vision распространяет его с оркестратор-прогонов на всю экосистему.

## §2 Целевая архитектура: три плоскости и шина событий

```
                    ВЛАДЕЛЕЦ
                       │
        ┌──────────────┼───────────────────────────┐
        │              │                           │
┌───────▼────────┐ ┌───▼───────────────┐ ┌─────────▼─────────┐
│  П1. МЕТОДО-   │ │  П2. РАНТАЙМ-СЛОЙ │ │  П3. CONTROL-PLANE │
│  ЛОГИЯ-КАК-    │ │  «pmo-core»       │ │  Кондуктор         │
│  ДАННЫЕ        │ │  (детерминизм)    │ │  (мультипликация)  │
│                │ │                   │ │  репо:             │
│ схемы артефак- │ │ lint · state ·    │ │  factory-conductor │
│ тов (JSON      │ │ gate · verify ·   │ │                    │
│ Schema) · FSM- │◄─┤ eval · budget    ├─►│ флот сессий ·      │
│ доктрины фаз   │ │                   │ │ очередь ready/     │
│ (YAML) · пра-  │ │ интерфейсы:       │ │ blocking · чекпойн-│
│ вила валидации │ │ CLI + MCP-сервер  │ │ ты · health ·      │
└────────────────┘ └───────┬───────────┘ │ бюджеты флота      │
                           │             └─────────┬─────────┘
                   ┌───────▼─────────────────────── ▼──────┐
                   │   ШИНА СОБЫТИЙ (единый контракт)      │
                   │   events.ndjson append-only · replay  │
                   │   bit-for-bit · materialized status   │
                   └───────────────────▲───────────────────┘
                                       │ читает/пишет через MCP/CLI
                          ┌────────────┴────────────┐
                          │  Claude Code — СЛОЙ     │
                          │  СУЖДЕНИЙ: discovery,   │
                          │  спеки, ресерч, ревью,  │
                          │  скиллы-методологи      │
                          └─────────────────────────┘
```

**П1 — методология-как-данные** (паттерн OpenSpec «методология = форкаемые данные»,
Spec Kitty «doctrine-as-data»). Канон артефактов и фаз выносится из прозы в
машиночитаемые схемы; markdown-скиллы/команды остаются носителями *суждений*, а их
структурные обязательства (frontmatter, поля, переходы) генерируются/проверяются от
схем. Прецеденты в репо уже есть: генераторы каталогов (`gen:*`), precedence
«код гейта > таблица > проза» (DEC-DEV-0197), check-validation-sync.

**П2 — детерминированный рантайм-слой `pmo-core`.** НЕ greenfield: генерализация
уже построенного оркестраторного ядра — `orchestrator/lib/fabric-engine.cjs`
(event-sourced, ndjson + materialized snapshot + bit-for-bit replay),
`autonomy-policy.cjs` (чистая gate-функция, урок 0201 «два guard'а — два вопроса»),
`run-ledger.cjs` (durable per-run + append-only ledger). Поверх — три чистые ниши
из coverage-check: eval скиллов, token circuit-breaker, MCP-интерфейс.
Стек: Node/CJS zero-deps (культура fabric-engine), тот же репозиторий,
npm-workspace-пакет. Никакого демона/БД/сервера — файловая истина + git, как сейчас.

**П3 — control-plane Кондуктор** — единая архитектура (решение владельца в интейке):
экосистема = методологическое ядро и правила, Кондуктор = рантайм мультипликации
сессий. Волны H0–H4 — SSOT в `dev/global-loop/PLAN.md`, физический код — в репо
`factory-conductor`. Вклад этого vision — donor-паттерны (Convoy-чекпойнты,
ready/blocking-очередь, TCB) адресуются ТУДА, в существующие волны, без второй
параллельной стройки.

**Шина событий — стержень единства.** Global-loop PLAN уже планирует `events.jsonl`
(Волна 3, H2 жёстко gated на него); fabric-engine уже пишет `events.ndjson` на
межпроцессном слое. Vision фиксирует: **контракт события один** — то, что пишет
продуктовый слой (`pmo state`), читает Кондуктор, и наоборот. Два формата = дрейф
по построению; проектирование контракта — первым шагом Волны 1 (PLAN T1.3).

**Интерфейсы pmo-core: CLI + MCP-сервер.** CLI — для гейтов/CI/хуков (как сейчас).
MCP — стратегический ход: методология становится доступна *любому* MCP-клиенту
(CC сегодня; другой харнесс, Кондуктор-сессии, headless-прогоны — завтра). Это даёт
межхарнессность **на уровне интерфейса** за копейки, откладывая дорогой полный EoS
(генерацию нативных команд под чужие форматы) до реального триггера.

## §3 Почему эта инфраструктура — «наиболее подходящая и надёжная»

Выбор от процессов и правил, не от моды (заказ владельца):

1. **Наши правила уже доказали форму.** Всё, что в экосистеме реально исполняется
   без надзора — исполняется детерминированным кодом (process-gate, check-counts,
   verify-цепь); всё, что прозой — дрейфует (уроки 0197, 0220). Инфраструктура,
   расширяющая первый класс — единственный ход, согласованный с эмпирикой репо.
2. **Ядро уже написано и оттестировано.** fabric-engine/autonomy-policy/run-ledger —
   23 файла тестов, live-валидация, побайтовая выживаемость скелета при `/compact`
   (SPEC §2-bis). Генерализация проверенного ядра надёжнее любого нового фреймворка.
3. **Zero-deps / файлы / git** — соло-дев с AI: ничего, что требует обслуживания
   (БД, демоны, облако), не проходит фильтр надёжности. Event-store = ndjson в git.
4. **MCP как интерфейс** — открытый стандарт, нативен для CC, покрывает будущие
   хосты; альтернатива (свой протокол) — изобретение велосипеда.
5. **Ресерч-обоснование:** единственные решения корпуса с shipped-детерминизмом
   (Spec Kitty FSM, MoAI-ADK линтер+хуки) — маленькие, но именно их механику
   воспроизводит рынок хуже всего; наша выжившая уникальность («тройка») усиливается
   ровно этим слоем. Рыночный минус (низкая аудитория строгих) нерелевантен:
   инструмент приватный, пользователь — мы сами.

## §5 Отвергнутые альтернативы

- **Полный уход с Claude Code** (ре-хостинг на Agent SDK / свой раннер): premise-check
  не подтверждает — слой суждений в CC силён, цена переезда огромна, выгода нулевая.
  Пересмотр — только при деградации самого хоста.
- **Standalone control-plane внутри экосистемы** (Autensa-класс: своё приложение,
  БД, GUI): дублирует Кондуктора и ломает решение 0216 о разделении домов; GUI при
  solo+терминал — фантомная аудитория (урок PHASE_D).
- **Greenfield pmo-core** «с чистого листа»: coverage-check показал готовое ядро;
  новый код поверх работающего — источник дрейфа, а не качества.
- **Немедленный полный EoS** (GSD-масштаб, ~16 рантаймов): цена L+ при отсутствии
  сегодня второго харнесса и решения о публикации; MCP-интерфейс закрывает 80%
  потребности на 20% цены.
- **Второй event-store** рядом с fabric-engine: два источника истины = дрейф по
  построению.

## §6 Анти-овер-инжиниринг рельсы

Наследуются рельсы `ECOSYSTEM_VISION.md` §6 целиком, плюс специфичные:
(1) каждый пункт PLAN проходит cuttable-scope; (2) ни одного нового долгоживущего
процесса/демона; (3) схема без потребителя не пишется (schema-first ≠ schema-only:
схема появляется вместе с первым чекером по ней); (4) FSM гейтит **переходы**, не
творчество внутри фаз — с escape-hatch класса `[skip-process-gate]`; (5) EoS не
начинается без триггера Ф4; (6) любое заимствование — через inspection brief
(borrow-not-adopt, прецедент bmad-loop DEC-META-0002), чужой код не тащим.

## §7 Риски

| Риск | Митигация |
|---|---|
| Двойная стройка (экосистема ∥ Кондуктор) при solo | Единый план: Ф3 = вклад в волны global-loop, не своя ветка стройки; последовательность волн |
| Экстракция ядра ломает оркестратор | Re-export-шим, 23 теста зелёные до/после, отдельный PR только на экстракцию |
| «Методология-как-данные» скатывается в переписывание всех md | Инкремент от схем валидации (потребитель уже есть — artifact-validate); скиллы не переписываются |
| FSM душит discovery-творчество | Гейты только на переходах фаз; draft-режим свободен (как quiet-draft у artifact-validate) |
| Пилот ломается миграцией | Канон CLAUDE.md §6: soft-миграции, absent == старое поведение, wipe-protection |
| Event-контракт разойдётся с Кондуктором | Контракт проектируется совместно (T1.3) ДО кода обеих сторон; фикстуры-обменники в тестах |
