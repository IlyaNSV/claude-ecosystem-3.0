# Module-delivery shakedown

**Когда применима:** первая «боевая» сессия в проекте вскоре после поставки нового модуля/фазы (напр. первая сессия, использующая Design-модуль после его релиза). Цель аудита — убедиться, что только что поставленный модуль реально работает на живом проекте.

**С чем сравнивать:** smoke-план поставленного модуля + его SPEC (аудитор читает их как ground truth по ссылкам из `baseline`/контекста) + процесс-каталог.

**Приоритетные критерии:** A–D, F относительно артефактов модуля.

**⚠ Честная пометка (Инкр.1):** сигнал `module_recently_shipped` — **best-effort** (эвристика по дате последней версии CHANGELOG в пределах 21 дня + использование модульных команд). При неточности класс деградирует к `design`/`feature-definition`/`mixed-uncertain`. Smoke этой рубрики — best-effort; точный timing-сигнал укрепляется в Инкр.2 (обогащение маркера git-HEAD/датой).
