# Session Audit — как это работает (гайд для человека)

> **Коротко:** механизм, который **сам** просматривает завершённые **продуктовые** сессии Claude Code в пилотных проектах, проверяет их против спецификаций экосистемы, копит находки и предлагает (но НЕ применяет) улучшения для самой Ecosystem 3.0.
>
> **Это D7-инструмент** (про разработку самой экосистемы) — живёт в `dev/meta-improvement/`, в пользовательские проекты не деплоится. Аудитит **только продуктовые сессии** (не сессии по доработке самой экосистемы).
>
> Инженерный дизайн: [`../_archive/session-audit-v2/SESSION_AUDIT_V2_DESIGN.md`](../_archive/session-audit-v2/SESSION_AUDIT_V2_DESIGN.md) (архивирован — механизм живой, дизайн-док исторический). Решения: DEC-DEV-0056 / 0057 / 0059.

---

## Зачем это нужно

Когда ты гоняешь продуктовые сессии в пилоте (`/product:feature`, `/design:start`, `/integrator:add`…), хочется знать: **прошла ли сессия так, как задумано методологией экосистемы?** Соблюдены ли P-RULE (Devil's Advocate при изменении инвариантов/бизнес-правил), целостность ссылок, frontmatter-конвенции, и как сессия повлияла на `.product/`. И если одна и та же проблема повторяется из сессии в сессию — это сигнал, что надо чинить **саму экосистему**, а не каждую сессию вручную.

Механизм делает это автоматически и доводит до **предложения патча**, которое ты принимаешь или отклоняешь.

## Конвейер (6 шагов)

```
[1] Сессия в пилоте завершается
      │  SessionEnd-хук пишет дешёвый маркер (id, проект, транскрипт) в audit-index.md → Pending
      ▼
[2] Триггер: ты запускаешь аудит (вручную или полу-авто через /loop)
      ▼
[3] КЛАССИФИКАЦИЯ (детерминированно, classify.js)
      • Зоны (мульти-лейбл): какие PMO-зоны затронула сессия — D1-discovery / D2B-behavioral /
        D2B04-design / D6-integrator. Сессия может затронуть НЕСКОЛЬКО зон сразу — это норма.
      • Mode: feature / fix / refactor / maintenance — модулирует строгость проверки.
      ▼
[4] АУДИТ (claude -p, session-audit.md)
      Читает спеки затронутых зон как «эталон» (как должна была пройти сессия) + процесс-каталог,
      строит таймлайн действий, сверяет. Пишет per-session отчёт в audit-reports/<uuid>.md.
      + EFFECT-PROBE: детерминированный замер, как сессия повлияла на .product/ (что изменила,
        корректно ли, что осталось недоделано).
      ▼
[5] ЖУРНАЛ (audit-journal.ndjson)
      Находки из отчётов копятся across сессий, ключ = (зона | проверка | артефакт | сигнатура).
      Одна и та же проблема в N сессиях накапливает счётчик. ≥3 = «системная».
      ▼
[6] СИНТЕЗАТОР ПАТЧЕЙ (patch-synth.js)
      Берёт системные кластеры → claude -p СКЕПТИЧЕСКИ перепроверяет (несколько независимых
      «адвокатов дьявола», по умолчанию «опровергнуть») → если находка выжила, пишет
      patch-candidate с предложением фикса. Ты решаешь [Y/N/E/D]. НИЧЕГО не применяется само.
```

## Откуда берётся «эталон корректности»

Не из жёсткого пошагового сценария (сессии не детерминированы — 3 модуля в одной сессии это нормально), а из **спецификаций экосистемы, подгружаемых под конкретную сессию**:

- **Зона** говорит, *какие спеки* — ground truth (напр. `D2B-behavioral` → `docs/product-module/SPEC.md` поток F.0–F.10 + `processes.md` P-RULE-01/02).
- **Mode** говорит, *насколько строго* (косметическая правка BR не требует Devil's Advocate; семантическая — требует).
- LLM-аудитор читает эти спеки и сверяет фактический таймлайн сессии.

Реестр зон — это просто `.md`-файлы в [`rubrics/`](rubrics/); добавить зону = добавить файл, код трогать не надо.

## Модель доверия: «машина находит — человек решает»

- Детерминированная часть (классификация, журнал, кластеризация, effect-probe) — **воспроизводима**.
- LLM-часть (аудит, скептик-проверка, синтез) — **только предлагает и интерпретирует**.
- **Никакого авто-фикса.** Синтезатор перед предложением патча прогоняет *adversarial-verify* (несколько скептиков, по умолчанию опровергают) — чтобы правдоподобные-но-ложные находки не попадали в патчи. Финальное решение всегда за человеком через `[Y/N/E/D]`.

## Как запускать

```bash
# Аудит накопленных сессий (универсальный режим, по зонам):
node dev/meta-improvement/scripts/audit-smoke.js --classify            # вся Pending-очередь
node dev/meta-improvement/scripts/audit-smoke.js --classify --session-id=<uuid> --force

# Полу-авто (пока Claude открыт) — держит аудит почти на автомате:
/loop 45m node dev/meta-improvement/scripts/audit-watch.js

# Журнал находок:
node dev/meta-improvement/scripts/audit-journal.js --rebuild --stats   # пересобрать + кластеры

# Синтезатор патчей:
node dev/meta-improvement/scripts/patch-synth.js --dry-run             # кластеры, без LLM
node dev/meta-improvement/scripts/patch-synth.js --cluster=D2B-behavioral:C   # один кластер
node dev/meta-improvement/scripts/patch-synth.js                       # все системные кластеры

# Приёмная сторона upstream-фидбэка (DEC-DEV-0097): подобрать UF-outbox пилота + FB-ledger +
# журнал аудита под единый контракт находки и развести по DEC-DEV (open / likely-ported / ported):
node dev/meta-improvement/scripts/feedback-intake.js --outbox <пилот>/.product/.upstream/feedback-outbox.md
node dev/meta-improvement/scripts/feedback-intake.js --ledger dev/ORCHESTRATOR_LIVE_RUN_FB_LEDGER.md --json
```

## Upstream-фидбэк: приёмная сторона (`feedback-intake.js`, DEC-DEV-0097)

Session Audit — это **один** из трёх upstream-источников «пилот → экосистема». Два других:
- **UF-outbox** — `/ecosystem:meta-feedback` в пилоте копит **системные** дефекты экосистемы в коммитируемый `.product/.upstream/feedback-outbox.md` (`UF-NNN`); это **приёмная** сторона того контура (DEC-DEV-0090 фаза-2);
- **FEEDBACK-JOURNAL / FB-ledger** — находки dogfood-прогонов Оркестратора (`FB-NNN` / `FB-LR-NN`).

`feedback-intake.js` — детерминированный консолидатор (capture-don't-fix, как и весь механизм): подбирает любой из источников, приводит к **единому контракту находки** (`{uid, source, source_ref, severity, signature, status, …}`) и разводит каждую против `DEV_JOURNAL.md`:

- **`ported`** — находка сама объявляет `DEC-DEV-NNNN` (в `Статус:`/route), и эта запись есть в журнале → решение принято, не пере-триажить;
- **`likely-ported`** — какая-то запись `DEC-DEV-*` **называет** её ref (или находка ссылается на чужой/отсутствующий DEC-DEV) → на ручную сверку (без тихой потери — наименовавший DEC-DEV выводится);
- **`open`** — ни то, ни другое → нужен upstream-триаж (→ `DEC-DEV-*` в этом репо).

Ничего не применяет и не редактирует (ни outbox, ни артефакты) — только читает, сверяет и докладывает. Приёмка (open → патч) остаётся ручным актом в репо экосистемы.

## Где что лежит

| Файл / папка | Что |
|---|---|
| `audit-index.md` | Очередь сессий: Pending (ждут аудита) / Processed (обработаны) |
| `audit-reports/<uuid>.md` | Per-session отчёт аудитора (находки + effect on product) |
| `audit-journal.ndjson` | Накопительный журнал находок (across сессий) |
| `patch-candidates/<zone>__<check>.md` | Предложения патчей, ждут `[Y/N/E/D]` |
| `rubrics/` | Zone-references (что и какими спеками проверять) |
| `prompts/` | Промпты аудитора и синтезатора |
| `scripts/` | classify · audit-smoke · effect-probe · audit-journal · patch-synth · audit-watch · **feedback-intake** |
| UF-outbox (в пилоте) | `.product/.upstream/feedback-outbox.md` — `UF-NNN`, источник для `feedback-intake` |

## Граница

Аудитятся **только owned-зоны** продуктового проекта (D1, D2-Behavioral вкл. дизайн, Integrator-handoff) — то, что реально видно в транскрипте Claude-сессии. Делегированные зоны (разработка, QA, выполняемые внешними инструментами через Integrator) механизму не видны — он видит только передачу в них.
