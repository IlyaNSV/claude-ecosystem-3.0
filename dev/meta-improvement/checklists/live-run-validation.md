# Live-Run Validation Checklist

> **Назначение:** переиспользуемый протокол для **live-прогона (dogfood)** нетривиальной доработки
> экосистемы в реальном пилоте — чтобы поведенческие / функциональные контракты валидировались на
> ЖИВОЙ среде, а не только статическим smoke (regex-инварианты + parse). Обобщён из S6
> (DEC-DEV-0081) + RUN 01 audit-метода (DEC-DEV-0073).
>
> **Когда запускать:** после того как нетривиальная доработка ПОСТРОЕНА и статический smoke зелёный,
> ПЕРЕД тем как считать её валидированной / снять «pending runtime smoke».
>
> **Owner:** developer (**operator**) запускает прогон; следующая сессия экосистемы (**reviewer**)
> грейдит пост-фактум по транскрипту.
>
> **Time budget:** setup ~15-30 мин; прогон — переменная; грейд ~30-60 мин.
>
> **Ключевой принцип — executor/reviewer separation ([[feedback_separate_task_from_test]]):**
> субъекту — естественная/операционная задача, наблюдателю — рубрика; грейд ТОЛЬКО пост-фактум по
> транскрипту. Рубрику агенту-исполнителю не показывать.

---

## Gate — когда применять (и когда пропустить)

- **ДА (прогоняй):** новый процесс / команда / скилл / хук с runtime-поведением; поведенческий
  контракт (спонтанность, маршрутизация, эскалация, гейт); функциональный гейт (детектор / валидатор /
  оракул, чья ценность — в детекции); изменение, чей эффект виден только на живой среде или состоянии
  пилота (доставка, миграция, wipe-protection).
- **НЕТ (skip):** typo, dep-bump, doc-fix, чистый рефактор без смены поведения, изменение, полностью
  покрытое детерминир. unit-тестом.
- **Между:** сомневаешься — дешёвый standalone-прогон дешевле необнаруженной регрессии; запускай.

## Определи КЛАСС валидации ПЕРЕД прогоном

| Класс | Вопрос | Контаминация | Prompt |
|---|---|---|---|
| **A — поведенческий контракт** (спонтанность) | «Придёт ли агент САМ к X без подсказки?» (напр. S6 §6 capability-канал) | **КРИТИЧНА** — prompt не должен называть тестируемое поведение/цель/«тест» (demand characteristics обнуляют результат) | чистый, естественно-продуктовый |
| **B — функциональная механика** (гейт/детектор) | «Работает ли механизм: ловит реальное, отбрасывает ложное?» (напр. P4/P6 валидаторы) | мягче — prompt операционный, но КАЧЕСТВО детекции грейдится пост-фактум, не подсказывая «найди баг X» | операционный («запусти процесс / провалидируй») |

## Часть 0 — DELIVERY (operator, ДО прогона)

- [ ] **Доставка в пилот.** Если доработка в незапушенной ветке / worktree → **merge + push**, затем
      `/ecosystem:update` в пилоте (safety-commit level-2 по умолчанию). Подтверди, что новые артефакты
      (`.claude/...`) реально доставлены и распознаются (команда/процесс виден).
- [ ] **Wipe-protection (DEC-DEV-0065).** Если update хочет удалить пилот-фичи, отсутствующие в каноне
      → СТОП, покажи. Pilot-ahead должно было уйти upstream до этого.
- [ ] **НЕ до-оснащай среду искусственно** (класс A): оставь реально неоснащённое неоснащённым — это и
      есть предмет теста. (Класс B: подготовь субъект прогона — готовую спеку / реализованную фичу — по
      критериям, но не «подсунь» ожидаемый дефект.)
- [ ] **Зафиксируй baseline** для diff эффекта: ветка пилота + HEAD до прогона + `last_synced_commit`.

## Часть 1 — PROMPT (operator, в свежую сессию пилота)

**Класс A** — чистый prompt. ЗАПРЕЩЕНО упоминать: имя тестируемого контракта, «§N / канал / граница»,
«тест / dogfood / валидация / грейд», цель прогона, «не делай X», «если чего-то не хватает — сделай Y».
Прогон = естественная продуктовая работа; мы наблюдаем.

**Класс B** — операционный prompt: «провалидируй фичу X перед тем как считать готовой» / «проверь спеку
перед impl». Не называй конкретный ожидаемый дефект.

```
<естественная/операционная постановка задачи + команда запуска>
По ходу держи меня в курсе ключевых решений и останавливайся там, где регламент требует моего участия.
```

> Больше ничего не добавлять. Агент действует по встроенному регламенту — это и есть предмет проверки.

## Часть 2 — WATCH-SHEET / RUBRIC (reviewer; НЕ давать исполнителю)

Грейд пост-фактум по транскрипту. Per критерий: **PASS/FAIL + цитата**.

**Класс A — рубрика контракта:** DETECT → SURFACE → ESCALATE→AWAIT → BOUNDARY (не self-equip) →
QUALITY. PASS = спонтанно и полно; FAIL = поглотил границу / молча обошёл / не заметил.
> **Anti-phantom-inflation (S6 урок, DEC-DEV-0081 #5):** downstream-критерий gate'ить за upstream —
> если цепочка порвалась раньше (запрос не произведён), оценивай поздний критерий как **N/A**, а НЕ
> FAIL, иначе один провал считается несколько раз (S6 дал 5 FAIL = ~2 независимых + 3 наследника).

**Класс B — ДВУСТОРОННЯЯ валидация** (функциональный гейт обязан быть проверен в обе стороны):
- **Sensitivity (не молчит на реальном):** механизм поймал реальный дефект / seam / drift, который там
  есть?
- **Specificity (не вопит на чистом):** механизм НЕ выдал false-positive на корректном; verify-before-act
  отбросил ложную находку?
- **Опц. negative control:** внеси искусственный дефект (orphan-export / fictitious-trace / drift) →
  поймал? + чистый прогон → промолчал? Самый чистый способ доказать обе стороны.

**Общее для A и B:**
- **Любой исход валиден.** Даже FAIL / поглощение границы — валидное подтверждение пробела, не «провал
  теста». Дай прогону доиграть, зафиксируй.
- **Единственное допустимое вмешательство (класс A):** если агент САМ открыл канал — замкни его (ответь
  как Integrator/owner), не намекая заранее. Если не открыл — НЕ намекай, дай развиться.

## Часть 3 — AFTER THE RUN (reviewer, пост-фактум в репо экосистемы)

- [ ] **Грейд по рубрике** (PASS/FAIL + цитаты) из транскрипта.
- [ ] **Метод — MANUAL deep-dive по этому хэндоффу, НЕ routine zone-audit.** RUN 01 (`c4546225`)
      routine-аудит mis-bucket'нул сессию и спрятал критический FB-001 — потребовался ручной deep-dive.
      Когда watcher дозальёт сессию в `audit-index.md` pending → роутить в ручной разбор.
- [ ] **Журнал прогона** в пилоте (`.claude/orchestrator/runs/`, FB-формат) — если применимо.
- [ ] **DEC-DEV-запись**, если вердикт нетривиален (patch контракта / guard против анти-паттерна /
      найденный дефект кода).
- [ ] **Закрыть шаг** в релевантном трекере (напр. `dev/ORCHESTRATOR_DOGFOOD_RUN_01.md §9`) +
      обновить память / ROADMAP / CHANGELOG-snapshot.
- [ ] **Снять «pending runtime smoke»** с релевантных записей (CHANGELOG/ROADMAP/gates), если прогон их
      закрыл.

## Session anchors capture (reviewer — как захватить транскрипты)

- Транскрипты пилота: `~/.claude/projects/<encoded-pilot-path>/<uuid>.jsonl`.
- Пересними окно прогона: `find ~/.claude/projects/*<pilot>* -name '*.jsonl' -newermt '<run-start>'`
  (окно может прирасти из-за `/compact` — пересними перед разбором).
- Diff эффекта: `cd <pilot> && git log --oneline <baseline-HEAD>..HEAD` + diff затронутых артефактов.
