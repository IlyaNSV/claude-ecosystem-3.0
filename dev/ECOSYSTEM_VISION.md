## 0. TL;DR (для занятого ревьюера)

- **Вектор владельца верный**, и внешний ресёрч это подтверждает почти дословно: экосистема уже интуитивно делает то, что peer-reviewed-исследования называют best practice (spec-driven chaining, детерминированные оракулы, verify-before-act, hard-approve гейты).
- Три идеи — это **три оси одного куба**, а не три независимых задачи:
  - **Step -0.9 = ось охвата** (как далеко вниз по D1→D6 дотягивается конвейер). Сегодня он сшит только до «feature-GO в dev-воркспейсе»; от CI/деплоя/QA-инфры/мониторинга **нет ни процессов, ни инструментов**.
  - **Step -0.8 = ось качества входа** (полнота/точность спеки D1-D2B). Это **рычаг с наибольшим плечом**: ошибки входа компаундируются вниз по конвейеру (METR/error-compounding).
  - **Step 0 = ось автономии** (укрупнение мелких человеко-гейтовых шагов в 5-8 автономных «крупных» + консилиум как подготовка решения вместо пользователя).
- **Три «тормоза» из ресёрча** (детали §4), которые надо зашить в дизайн с первого дня:
  1. Completeness-loop окупается **только bounded + externally anchored** (cap итераций + judge-порог + convergence-Δ). Чистая само-критика выходит на плато к 2-3 раунду и **может деградировать**.
  2. Консилиум выигрывает для **judging/parallel-поиска**, но **не для одного связанного решения** (там он ~15× дороже и склонен к groupthink) — если только агенты не **по-настоящему гетерогенны** и панель работает как **жюри**, а не консенсус-дебаты.
  3. Полностью автономный «идея→прод» в 2026 **нереалистичен** (0.9⁷ ≈ 48% сквозной надёжности). Реалистичная цель — **100% покрытие пути с гейтами на границах фаз**, а не 100% автономии. Это переформулировка «100% результата».
- **Автономность — настраиваемый слой, не флажок (Epic F, по запросу владельца):** фиксированный enum уровней `L0 Assisted → L1 Supervised → L2 Consilium-gated → L3 Autonomous`; **детерминированный resolver** решает диспозицию гейта (`auto/consilium/human/block`) кодом, а не суждением; **floor необратимости** (prod-deploy/destructive/spend-money/secret) — non-crossable даже в L3 без отдельного opt-in; hard-configurable дефолт в `product.yaml` (применяется в каждой сессии) + параметризованный override на процесс/связку. Два именованных профиля владельца = `default`(L1) и `autonomous`(L3, задел на будущее с консилиум-гейтом вместо человека).
- **Рекомендуемый порядок:** сначала качество входа (Epic B) + фундамент профильных агентов (Epic A) + ядро autonomy-слоя (F1: resolver+config+L0/L1+override; *F1-core независим, но его wiring к gate-контракту — coordinate-only с оркестратор-сессией*), потом автономия (C/D + F2 консилиум-гейт), и только потом дотягивание до прода (E+F3, coordinate-only). Инвестировать во фронт первым — потому что компаундирование ошибок делает точность входа самым дешёвым местом для борьбы за качество.

## 1. Разбор интенций (что именно просят — точно и неизбыточно)

Идеи владельца записаны как «вектор», местами избыточно/неточно. Ниже — переформулировка в исполнимые намерения. Где формулировка содержала переусложнение — помечено.

### Step -0.9 → **Intent A: связность до прода**
> «Полная связность процессов, чтобы прийти в точку *Приложение запущено на проде и полностью функционально*.»

**Точное намерение:** закрыть разрывы конвейера так, чтобы существовал **непрерывный путь** (процесс + инструмент + контракт) от `handoff.md` до работающего на проде приложения, с явными гейтами там, где это необратимо/дорого.
**Корректировка:** «полностью функционально на проде» **автономно и со 100% результата** — недостижимо в 2026 (§4 кластер 3). Достижимо: **полное покрытие пути** (нет «дыр-процессов») + **gated-автономия**. Цель = «нет ни одного шага, который некому исполнить», а не «человек ни разу не нужен». *Честная оговорка: покрытие пути гарантирует отсутствие осиротевшего шага — НЕ гарантирует исправный артефакт на проде; качество результата всё равно зависит от исполнителей (агентов/человека) каждого покрытого шага.*

### Step -0.8 → **Intent B: bounded completeness-loop + профильные агенты**
> «Механизм *полной завершённости* описания продукта (D1-D2B) через loop, волнами, с обращением к профильным субагентам (UI/UX, SEO, …) для решений по неоднозначностям. Граница завершённости — полезно для точности всех ИИ-агентов, но не избыточно. Полный аудит процессов под loop-режим. Синергия профильных агентов: гарантировать, что их зовут, когда процесс затронул их зону.»

**Точное намерение, три под-намерения:**
- **B1 — completeness-loop:** итеративно доводить артефакты D1-D2B до «достаточной» полноты, авто-разрешая разрешимое и эскалируя по-настоящему неоднозначное.
- **B2 — граница «достаточности»:** операционализировать «полезно, но не избыточно». **Ответ ресёрча + кода:** граница = **handoff DoR / rubric того, что реально нужно downstream-агентам** (cc-sdd/impl), а не субъективное «идеально». Стоп-условие — bounded (§4 кластер 1).
- **B3 — профильные агенты + zone-routing:** реестр гетерогенных персон-агентов и **детерминированный роутер «зона процесса → агент(ы)»**, чтобы профильный агент *гарантированно* срабатывал, когда шаг затронул его зону. «Не бояться разбить процессы на зоны» = декомпозировать шаги с тегами зон.

### Step 0 → **Intent C: крупные автономные шаги + консилиум + ветвление**
> «Доделать фичу с *крупными* работами (например *полное обогащение feature set*) + сгруппировать шаги продуктовых процессов в *крупные* так, чтобы автономно дойти от скелета до MVP за 5-8 крупных шагов. Тщательно проработать *консилиум* (продуктовая группа, архитектурный отдел, тимлиды+команды, QA-отдел) как замену пользователя для оптимальных решений. Полный релиз со сложным маршрутом, предусмотреть кейсы >10% вероятности, зашить в agent-loop для 100% результата, мб внешний инструмент. Использовать самые актуальные инструменты (layer loops, agent teams, workflow) и оптимальные паттерны.»

**Точное намерение, под-намерения:**
- **C1 — крупные work-units:** макро-процессы, агрегирующие десятки мелких шагов (пример: batch-обогащение всего feature set одним прогоном). 5-8 «крупных шагов» = 5-8 **границ фаз с гейтами** (что совпадает с рекомендацией ресёрча — гейтить на границах фаз).
- **C2 — консилиум:** многоперспективная подготовка решений в точках, где сейчас зовётся пользователь / конфликт / brainstorm.
- **C3 — ветвление (>10% кейсы):** предвосхищать высоковероятные ветки и зашивать их в детерминированный скелет Workflow (известная ветка → авто-обработка; новое/редкое → эскалация).
- **Корректировка:** «консилиум как *замена* пользователя для *оптимальных* решений» — опасная крайность (§4 кластер 2). Безопасно: консилиум **готовит** решение (опции, steelman, pre-mortem) → дешёвый человеко-гейт; авто-решает только для обратимых/дешёвых; для связанных/необратимых — информирует человека, не заменяет.

### Сквозная тема
**Профильные агенты (B3)** и **консилиум (C2)** опираются на один и тот же фундамент — **реестр гетерогенных персон-агентов**. Поэтому он выделен в отдельный Epic A.

## 3. Целевая модель (как три идеи складываются в одну траекторию)

```
                 ось АВТОНОМИИ (Step 0)
                         ▲
                         │   крупные шаги (C1) ── консилиум (C2/D) ── ветвление (C3)
                         │        ▲
   ось КАЧЕСТВА ВХОДА    │        │ опираются на
   (Step -0.8) ──────────┼────────┴── фундамент: профильные агенты + zone-routing (A)
   completeness-loop (B) │
   bounded + DoR-anchor  │
                         │
                         └────────────────────────────────────────────►  ось ОХВАТА (Step -0.9)
                              D1 ─ D2B ─ [handoff] ─ D2T ─ D3 ─ │разрыв│ ─ D3-04…D5 ─ ПРОД
                                            ✅ сшито до сюда            ❌ Epic E (coordinate)
```

Логика связки и **почему такой порядок**:
1. **A (профильные агенты)** — общий субстрат для B и C2/D. Дёшево, реюзает паттерн DA. Строим первым.
2. **B (completeness-loop)** — поднимает качество входа. Это **рычаг №1**: по METR/error-compounding ошибка спеки множится вниз по всем downstream-шагам; чинить дешевле всего у источника. До расширения автономии вниз (C/E) — фронт должен быть точным.
3. **C/D (крупные шаги + консилиум)** — автономизируют исполнение поверх качественного входа; консилиум готовит решения в scope-точках.
4. **E (до прода)** — последним и coordinate-only: зависит от §6-фикса Оркестратора + установки Интегратором D3/D4/D5-инструментов.

## 4. Внешний ресёрч — profit vs переусложнение (cited, актуально на сер. 2026)

> Дисклеймер целостности: часть поисковых выдач упоминала пост-cutoff модели/версии/arXiv-ID (после янв-2026) — они помечены *unverified*. Несущие утверждения опираются на primary-источники 2023–янв-2026.

### Кластер 1 — completeness / reflection loops (→ Epic B)
- **Паттерн:** evaluator-optimizer / generator-critic / reflection. Инстансы: **Self-Refine** (~20% средн. прирост на бенчмарках, где помогает — task-dependent; gains в **первых 1-2 раундах**, дальше плато), **Reflexion** (опирается на **внешний reward**, не чистая саморефлексия).
- **Контр-доказательство:** **Huang et al. «LLMs Cannot Self-Correct Reasoning Yet»** (arXiv 2310.01798, ICLR 2024) — без внешнего сигнала само-коррекция «иногда **ухудшает** результат»; многие про-результаты тайком используют **ground-truth как стоп-оракул** (которого в проде нет).
- **Стоп-критерий (ответ на B2):** комбинировать **(i) hard max-iter cap + (ii) judge/score-порог τ + (iii) convergence Δ<ε**, желательно с *внешним* сигналом (rubric/judge, или information-gain→0), не само-оценкой генератора. Принципиальный «спрашивать vs продолжать» = **Active Task Disambiguation** (arXiv 2502.04485): выбирать вопрос, **максимизирующий information gain**, и **стоп когда info-gain→0**. *(Совпадает с ITP владельца — та же логика information-gain.)*
- **Тулинг:** Anthropic предписывает «stopping conditions (max iterations)»; LangGraph `recursion_limit` (default 25); CrewAI `max_iter` (~20-25) + Task Guardrails до критериев/`max_retries`; сообщество редко видит улучшение после ~3 reflection-итераций.
- **ВЕРДИКТ:** *Profitable, но только bounded + externally anchored.* 1-3 итерации с judge-порогом и convergence-чеком — высокая ценность для доводки спеки. Открытый само-критик-loop без внешнего сигнала — это и есть ловушка переусложнения: плато к раунду 2-3, риск деградации, жжёт токены.

### Кластер 2 — консилиум / совет / дебаты (→ Epic C2/D)
- **Паттерны:** Multi-Agent Debate (MAD), **LLM-as-jury (PoLL)**, **Mixture-of-Agents (MoA)**, role-crews, supervisor/hierarchical, swarm/handoffs.
- **Про-совет (для *судейства/агрегации/параллельного поиска*):** **PoLL** (arXiv 2404.18796) — жюри малых **разных** моделей бьёт один большой judge, **~7× дешевле**, меньше bias. **MoA** (2406.04692) — 65.1% vs GPT-4o 57.5% (AlpacaEval 2.0). Anthropic multi-agent research (2025-06) — **+90.2%** vs single-agent (но это **breadth-first параллельный поиск**, не совещающийся совет).
- **Скептик (для *связанных решений*):** **«Stop Overvaluing Multi-Agent Debate»** (arXiv 2502.08788) — MAD «часто **не превосходит** single-agent CoT/Self-Consistency при кратно большей цене»; фикс — **гетерогенность моделей, а не больше агентов**. ICLR 2026 (Revisiting MAD) — MAD помогает в основном для **судейства/safety**, не для solution-finding. Groupthink/sycophancy — задокументированный failure mode гомогенных советов.
- **Цена (Anthropic, primary):** агенты ~4× токенов чата; **мульти-агент ~15×** *(цифра — для breadth-first research-сетапа Anthropic; для совещания об одном связанном решении точно не измерена, но порядок тот же)*. Anthropic прямо: мульти-агент **плохо подходит** для «доменов, где всем агентам нужен общий контекст и много зависимостей» — а это **ровно** случай совета, спорящего об одном связанном решении.
- **Тулинг:** CrewAI (`Process.hierarchical`, `manager_agent`); AutoGen/AG2 (`GroupChat` + `speaker_selection`); LangGraph (`langgraph-supervisor` vs `langgraph-swarm`); OpenAI Agents SDK (`Handoffs`, `Guardrails`).
- **ВЕРДИКТ:** *Profitable для судейства/агрегации/параллельного поиска; переусложнение для одного связанного дизайн-решения.* Консилиум-как-замена-человека **не** бьёт надёжно один сильный модель-вызов с хорошим промптом на связанном решении, стоит порядка ~15× (по аналогии с breadth-first; точная цифра для совещания не измерена), рискует groupthink — **если** не обеспечить реальную гетерогенность и не трактовать совет как **жюри/параллель**, а не консенсус-дебаты.

### Кластер 3 — автономный pipeline идея→MVP (→ Epic C1/A/E)
- **Паттерны:** orchestrator-workers, prompt-chaining, routing, parallelization, evaluator-optimizer (5 паттернов Anthropic); plan-and-execute; доминирующий 2026-паттерн — **spec-driven autonomous coding** (GitHub Spec Kit, AWS Kiro, Devin-style). *Экосистема это уже делает через cc-sdd/kiro.*
- **Реализм (METR, 2025-03-19):** длина задачи при 50% надёжности удваивается ~каждые 7 мес, **но горизонт 80%-надёжности в 4-6× короче 50%-горизонта**. Для цепочки 5-8 шагов даже 90%/шаг → **0.9⁵–0.9⁸ ≈ 59%→43%** (7 шагов ≈ 48%) сквозной надёжности. **Полностью автономный идея→прод нереалистичен.** Автономия высока *внутри* хорошо-специфицированного шага, низка *поперёк* цепочки.
- **Failure modes:** error-compounding/cascade, «agent meltdown» (зацикливание/самопротиворечие/галлюцинация tool-output на длинном горизонте), галлюцинированные API/схемы, integration/deploy («verification tax»). Бенчмарки скептичны: Devin ~13.9% (vendor) vs независимо ~3/20; контаминация SWE-bench → высокие % = верхние оценки.
- **HITL-тулинг:** LangGraph `interrupt()` (пауза в узле, чекпойнт, resume `Command(resume=...)`) — **caveat: на resume узел перезапускается с начала → pre-interrupt side-effects повторяются → проектировать идемпотентно** (= ровно та idempotency-дыра, что Оркестратор нашёл в §2-bis RUN 01). Гейтить **выборочно, на границах фаз** (spec-approval, plan/design sign-off, pre-merge, pre-deploy/high-privilege — ровно швы Spec Kit/Kiro). Over-gating → fatigue; under-gating → cascade. OWASP Agentic — гейтить high-privilege действия.
- **ВЕРДИКТ:** *Profitable per bounded step, переусложнение при погоне за end-to-end unattended.* Реалистичный 2026-дизайн = spec-driven chaining + orchestrator-workers с **человеческими гейтами на границах фаз**. Полностью автономный 5-8-шаговый идея→MVP — known failure mode, не stretch-goal.

## 5. Дорожная карта — эпики, инкременты, зависимости

> Каждый эпик: намерение → инкременты → где живёт → anti-over-engineering рельсы → вердикт. Оценки часов **не даю** (эмпирический множитель ROADMAP ×2-4 ненадёжен на R&D-задачах); даю **порядок и зависимости**.

### Граф зависимостей
```
A (профильные агенты + zone-routing)  ──┬──►  B (completeness-loop)
                                         └──►  D (консилиум-примитив)
F1 (autonomy resolver + config + L0/L1 + override)  ── рядом с A, независимо
B ──► C (крупные шаги)  ◄── D (консилиум как гейт ВНУТРИ C)
D ──► F2 (L2/L3 консилиум-гейт)
C / D / F2 ──► (питают) ──► E (+F3, до прода, coordinate-only: Orchestrator §6 + Integrator D3/D4/D5)
```
**Рекомендуемая последовательность:** (A ∥ F1) → B → (C1 ∥ D) → F2 → E(+F3). F1 (детерминированный слой настройки + L0/L1 + параметризация) можно строить рано и независимо — консилиум-гейт (L2/L3) ждёт D.

### 5.1. Как эпики ложатся на текущую траекторию экосистемы (для решения #7)

Текущая траектория: Product/Design-модули (D1-D2B) **отгружены** (Phase 2-6); фронт активной работы — **параллельная оркестратор-сессия** (D2-T+/runtime: сейчас на gate-contracts, P5 T5 / P6, до deploy/P7 ещё далеко) + цель PILOT POINT. Куда садятся новые эпики:

| Эпик | Зона/модуль | Отношение к траектории | Коллизия с оркестратор-сессией | Старт сейчас? |
|---|---|---|---|---|
| **A** персоны+zone-routing | Product/Design (owned) | расширяет отгруженные Phase 2-6 | нет (другая зона) | ✅ да, малый (клон DA) |
| **B** completeness-loop | Product (owned) | расширяет P2; рычаг качества входа | нет | ✅ да (полная волна ждёт A; минимальная B на DA — можно раньше) |
| **C** крупные шаги | Product macro (Workflow) | новый слой поверх P2 | низкая (общий Workflow-движок, зона D1-D2B ≠ D2-T) | после B |
| **D** консилиум | cross-cutting | реюз §7.6 + Workflow `parallel()` | низкая | с/после A |
| **F1** autonomy resolver+config | cross-cutting | **смыкается с тем, что оркестратор-сессия делает ПРЯМО СЕЙЧАС** (gate-risk-classifier, env-readiness, verify-finding, gate-contracts) | ⚠ **точка синергии / риск дубля** — резолвер потребляет risk-tier, который у них уже считается | ✅ да, **но согласовать** общий контракт гейтов |
| **F2** консилиум-гейт | cross-cutting | после D | — | после D |
| **E** до прода | Orchestrator+Integrator (delegated) | = трек оркестратор-сессии; gated их работой + установкой Интегратором D3/D4/D5 | **полностью их зона** | нет (далеко; см. #6) |

**Вывод (рекомендация по #7):** фронт-эпики **A / B / F1 не конфликтуют** с параллельной сессией (owned Product/Design + cross-cutting) и дают наибольший рычаг рано. Рекомендуемый первый инкремент: **Epic A (минимум: `architect`/`qa`/`ux` персоны + zone-routing)** → сразу **минимальная Epic B на пилоте `my-first-test`** (по принципу «dogfood после каждого инкремента», CLAUDE.md §2). **F1** — параллельным независимым треком, но его контракт гейтов **согласовать с оркестратор-сессией** (иначе два расходящихся механизма gate-policy). C/D/F2/E — позже по графу. Это «ложится» на траекторию без коллизий: фронт пайплайна двигаем здесь, хвост (E) остаётся у оркестратор-трека.

### Epic A — Профильные агенты + zone-routing *(фундамент; Product/Design-owned)*
**Намерение:** B3 — реестр гетерогенных персон-агентов + детерминированный роутер «зона процесса → агент», чтобы профильный агент гарантированно срабатывал в своей зоне.

**Инкременты:**
- **A1.** Стартовый набор персон (клон конвенции DA), приоритет — те, что мапятся на существующие PMO-зоны и downstream-нужды: `architect-advisor` (D2-T01/T02 предвестник), `qa-advisor` (D4/acceptance, питает VC/IC), `ux-advisor` (D2-B04/Design), плюс по нужде `security-advisor`, `data-advisor`, `seo-advisor`. Каждый — отдельный `subagent_type`, structured-verdict schema, adaptive-depth (как DA: cosmetic→quick / significant→full), anti-pattern список.
- **A2.** `zone → agent` манифест (аналог `pmo-mapping.yaml`, но agent-side): какой шаг какие зоны затрагивает → какие персоны firing. Детерминированный роутер (`.cjs` оракул, не суждение) поверх паттерна DA-хука: хук детектит изменение в зоне → пишет `*-pending.yaml` → процесс спавнит matching-персону.
- **A3.** Применить `da-subagent-type-contract` ко всем персонам (canonical-type-always, no silent fallback → loud error).

**Anti-over-engineering рельсы:** (1) персона firing **только** при затронутой зоне **и** magnitude ≥ порога (adaptive-depth) — иначе панель сожжёт токены на тривиальностях; (2) гетерогенность — каждая персона несёт *отличный* prior (иначе groupthink, §4 кластер 2); (3) старт с 3 персон, расширять по факту пробелов, не спекулятивно.
**Вердикт:** низкий риск, высокий рычаг (субстрат для B и D). Реюз DA-паттерна делает это дешёвым.

### Epic B — Bounded completeness-loop для D1-D2B *(Product-owned)*
**Намерение:** B1+B2 — итеративно доводить спеку до «достаточной» полноты; авто-разрешать разрешимое, эскалировать неоднозначное; граница = DoR/rubric, стоп = bounded.

**Инкременты:**
- **B1.** `completeness-oracle` — детерминированный слой (расширение `validation.md` + покрытие SC/BR/LC/IC/VC + handoff DoR) + LLM-judge против rubric per artifact-type. Считает «completeness score» и список gaps/ambiguities. **Граница «достаточности» (B2) = score ≥ τ, где τ калибруется по тому, что реально потребляют cc-sdd/impl-агенты** (handoff DoR — уже существующий якорь).
- **B2.** Волновой loop: wave = (профильные персоны Epic A проходят по артефактам) → (gaps классифицируются: разрешимо-автономно vs нужен decision) → (разрешимое — авто-фикс с verify-before-act; decision — в очередь эскалации). **Стоп:** max-waves cap **И** (score ≥ τ **ИЛИ** Δscore < ε между волнами **ИЛИ** info-gain открытых вопросов → 0).
- **B3.** Эскалация: разрешимое/обратимое → авто (T1-стиль ITP, фиксировать допущение); связанное/стратегическое → консилиум (Epic D) готовит → человеко-гейт. Многоходовый clarify-loop при отклонённом гейте (re-formulation, как нашёл Orchestrator §7-bis).
- **B4.** Аудит процессов под loop-режим (явная под-задача из Step -0.8): пройтись по P1/P2/P2.5, пометить какие шаги loop-safe, какие требуют идемпотентности/гейта (учесть idempotency-дыру §2-bis Оркестратора). Выход — таблица «шаг → loop-safe? → нужный предохранитель».

**Anti-over-engineering рельсы:** (1) **никогда** не само-оценка генератора как единственный стоп — всегда внешний judge/oracle (§4 кластер 1, Huang et al.); (2) 1-3 волны типично, hard-cap; (3) τ привязан к downstream-нуждам, не к «идеалу» — это и есть ответ «полезно, но не избыточно»; (4) loop пишет, что осталось недо-разрешённым (no silent truncation).
**Вердикт:** *высокая ценность при дисциплине bounded.* Рычаг №1 (качество входа гасит компаундирование). Главный риск — соблазн сделать loop открытым; рельсы выше его снимают.

### Epic C — Крупные автономные шаги *(Product-owned; реюз Orchestrator Workflow-паттерна)*
**Намерение:** C1+C3 (под-намерения из §1) — сгруппировать F.1-F.10 / D1.x в 5-8 крупных work-units; «полное обогащение feature set» одним прогоном; зашить >10%-ветки в скелет. *(Под-намерение C2 «консилиум» реализуется в Epic D — здесь не дублируется. Инкременты ниже помечены C-i/ii/iii, чтобы не путать с под-намерениями C1/C2/C3 из §1.)*

**Инкременты:**
- **C-i** (реализует C1). Макро-процесс `batch-enrich-feature-set` (Workflow `pipeline()` по FM релиза; каждый FM проходит стадии F.2→F.10; гейт на границе фаз, не на каждом шаге). Это «крупная работа» из примера владельца.
- **C-ii** (структура C1). Определить 5-8 «крупных шагов» от скелета до MVP-handoff как **границы фаз** (= точки гейта; кто его закрывает — человек / консилиум / auto — задаёт уровень Epic F, по §4 кластер 3). Черновик границ: [1] feature-set enrichment → [2] design-pass (has_ui) → [3] cross-feature consistency/cascade → [4] adversarial/DA + completeness-loop (Epic B) → [5] handoff-batch. (D2-T/D3 крупные шаги — уже Orchestrator P3/P5.)
- **C-iii** (реализует C3). Branch-anticipation: per-макро-шаг перечислить ветки с вероятностью >~10%, зашить известные в детерминированный control-flow Workflow (известная ветка → авто; новое/редкое → эскалация в B3/D). Явно `log()` отброшенные/неучтённые ветки.

**Anti-over-engineering рельсы:** (1) автономия **внутри** шага, гейт **на границе** (METR: поперёк цепочки надёжность падает); (2) идемпотентность шагов (LangGraph resume-caveat = idempotency-дыра Оркестратора); (3) не гнаться за «0 гейтов» — 5-8 границ это и есть гейты; (4) переиспользовать durable Workflow-скелет (переживает /compact), не материализовать continuation вручную (урок RUN 01).
**Вердикт:** *profitable per bounded step.* Эмпирически частично доказано (RUN 01: 23/26 без гейта). Риск — погоня за end-to-end unattended; держать гейты на границах.

### Epic D — Консилиум-примитив *(cross-module; реюз §7.6 + Workflow `parallel()`)*
**Намерение:** C2 — переиспользуемый «совет» для точек решений/конфликтов/brainstorm, как подготовка решения, не замена гейта.

**Инкременты:**
- **D1.** Консилиум-примитив = `parallel()` фан-аут на N **гетерогенных** персон (Epic A) с **разными priors** → synthesis-агрегатор → structured-verdict → **в гейт** (не вместо). Реализован как **жюри/breadth-first**, не консенсус-дебаты (§4 кластер 2). Опционально — разные модели для усиления гетерогенности.
- **D2.** Политика (реюз §7.6): консилиум легитимен **только при объявленном scope** (subject + prior-labels + comparison-axes) + сохранённый approve-gate; **no silent fan-out**; на связанном/необратимом — **информирует**, не решает.
- **D3.** Adaptive-depth/cost-gate: консилиум (дорогой, ~15×) firing **только** для high-stakes/связанных решений; для дешёвых/обратимых — одиночная персона или авто (как DA cosmetic/significant). «Продуктовая группа / архитектурный отдел / QA-отдел / тимлиды» = **именованные пресеты panel-состава** под класс решения.

**Anti-over-engineering рельсы:** (1) жюри, не дебаты; (2) гетерогенность обязательна (иначе groupthink + лишние ~15× токенов без выигрыша); (3) cost-gate — не звать совет на всё подряд; (4) **никогда** не auto-decide связанное/необратимое — это против анти-автономного зерна кода и против §4 кластер 2.
**Вердикт:** *profitable как jury/подготовка решения; переусложнение если станет консенсус-дебатами или авто-заменой человека.* Самая опасная идея по риску — рельсы критичны.

### Epic E — Сегмент конвейера до прода *(coordinate-only; Orchestrator + Integrator-owned)*
**Намерение:** A — D3-04…D5: CI/build, provisioning, deploy/rollback, QA-инфра, monitoring.

**Состояние и предусловия (НЕ реализую здесь — зона параллельной сессии):**
- **Предусловие 1:** Orchestrator §6 capability-канал починен (S6 = FAIL сейчас) — без него Оркестратор не может «запросить руки/голову» для деплоя.
- **Предусловие 2:** Integrator устанавливает первые D3/D4/D5-инструменты (CI/CD, deploy-таргет, test-infra, observability) — их сейчас нет.

**Anti-over-engineering / честность цели:** цель E = **полное покрытие пути с гейтами на границах** (pre-deploy, prod = human-gate per autonomy-tiers), **не** 100% unattended (§4 кластер 3 + OWASP high-privilege gating). «100% результата» владельца → переформулировать как «100% покрытия пути». Формальный handoff-work-order спинуть, когда оркестратор-трек реально дойдёт до deploy/P7.
**Вердикт:** *необходимо для Intent A, но это самостоятельный модульный трек, не доработка.* Лежит в зоне параллельной сессии → **координировать, не строить в этом доке.**

### Epic F — Уровень настройки автономности (autonomy configuration layer) *(cross-cutting; Product-config + Orchestrator-runtime)*
**Намерение (новое, запрос владельца):** детерминированный и **жёсткий** механизм управления автономностью: (1) два+ именованных профиля — `default` (человеко-гейты) и `autonomous` (скелет→прод, консилиум-гейт вместо человека; *сейчас risky/raw — задел на будущее, где агенты ближе к человеку в проф-решениях*); (2) hard-configurable дефолт, применяемый **прямо сейчас в каждой сессии проекта**; (3) **параметризованные вызовы** — команда на полную автономию процесса или их связки поверх дефолта; (4) гибкий уровень при сохранении детерминизма и безопасности.

**Почему отдельный слой, а не флажок:** автономность пересекает Epic C (крупные шаги), D (консилиум = подменный гейт), E (до прода) и существующие autonomy-tiers (orchestrator §7) + `env_tiers`. Вынести в один детерминированный resolver — иначе политика расползётся по процессам (drift).

**Модель — фиксированный enum уровней** (детерминизм: enum, не континуум), маппится на две существующие оси (review-level 🔴🟠🟡🟢 × env-tier dev/staging/prod):

| Уровень | Кто закрывает 🟠/🔴-гейт | Auto | Подмена человека |
|---|---|---|---|
| **L0 Assisted** | человек на всех 🟠+🔴; auto только 🟢 | минимум | нет |
| **L1 Supervised** *(дефолт, ≈ реальность RUN 01)* | человек на необратимом/staging+; auto на обратимом/dev | широко | нет |
| **L2 Consilium-gated** | консилиум (Epic D) до floor; человек только выше floor | да | консилиум-жюри + порог |
| **L3 Autonomous** *(задел на будущее, risky/raw)* | консилиум сквозь до прода; floor по умолчанию **неизменен** (сужение — только отдельным opt-in) | макс | консилиум вместо человека |

**Floor (предохранитель необратимости) — ключ к «risky, но ответственно».** Набор operation-классов, которые **всегда человек**, независимо от уровня и override — пока сам floor не переконфигурирован отдельным явным opt-in. Дефолт floor = `{prod_deploy, destructive/data-loss, spend_money, provision_real_secret/external_account}`. Маппится на orchestrator §6-bis provisioning-tier + §7 prod-gate. → даже `L3` по дефолту **НЕ** деплоит на прод вслепую.

**Детерминированное разрешение** (pure-function resolver `lib/autonomy-policy.cjs`):
```
floor (non-crossable, пока floor не переконфигурирован)
  > per-invocation override (--autonomy=…)
  > per-process pin (product.yaml process_overrides)
  > project default (product.yaml autonomy.default_level)
  > ecosystem built-in (L1)
```
`resolve(operation_class, risk_tier, env_tier, policy, override) → disposition ∈ {auto | consilium-gate | human-gate | block}`. Чистая функция, тестируемая, same-inputs→same-disposition. **Единственное LLM-суждение — содержание вердикта консилиума; само решение о диспозиции — код.** Это и есть «детерминированно и жёстко».

**Консилиум-гейт с safe-fallback (L2/L3):** консилиум (Epic D, гетерогенное жюри) → structured-verdict + confidence; `confidence ≥ τ → proceed`, иначе → **human fallback** (не слепой auto). «Агенты вместо человека» деградирует безопасно, а не падает в прод.

**Config (hard, schema-validated, backward-compat — absent == L1, прецедент `product_class`):**
```yaml
autonomy:                      # .claude/product.yaml — применяется в каждой сессии проекта
  default_level: L1
  floor: [prod_deploy, destructive, spend_money, provision_real_secret]
  consilium_gate: { confidence_threshold: 0.8, panel: architecture }
  process_overrides: { deploy-to-stage: L0 }   # пин: не выше L0 даже в L3-сессии
```
**Параметризованный вызов:** `/orchestrator:run <process> --autonomy=L3` (один процесс) или `--autonomy=autonomous` на связку; override распространяется по цепочке, но **не пересекает floor** и не поднимает запиненные процессы.

**Locked (2026-06-23, владелец принял):** enum `L0/L1/L2/L3`; профили `default`=L1 / `autonomous`=L3; дефолт новых проектов = **L1**; floor по умолчанию = `{prod_deploy, destructive, spend_money, provision_real_secret}`; override-флаг `--autonomy=` на `/orchestrator:run` **и** на продуктовых макро-командах (Epic C); гранулярность = процесс/связка + `process_overrides`-пины (без per-шаг override).

**Инкременты:**
- **F1 (рано, рядом с A):** resolver `autonomy-policy.cjs` + config-схема + L0/L1 + параметризованный override + **audit-trail** каждой диспозиции (replayable). Уже даёт «детерминированную настройку дефолта + параметризованные вызовы» без консилиума.
- **F2 (после D):** L2/L3 — консилиум-гейт с порогом + human-fallback; floor-механика; именованные профили `default`/`autonomous`.
- **F3 (с E):** распространение на prod-сегмент (floor реально упирается в deploy).

**Anti-over-engineering / safety рельсы:** (1) **floor non-crossable** без отдельного opt-in — «полная автономия» не должна *случайно* деплоить на прод; (2) консилиум-гейт = жюри+порог+human-fallback, не слепой auto (§4 кластер 2); (3) **reversibility-first** в L2/L3 (feature-flags, staging-first, safety-commit DEC-DEV-0061) — ошибочное авто-решение восстановимо; (4) полный audit-trail (детерминизм + replay); (5) дефолт остаётся L1, выше — явный opt-in (backward-compat); (6) enum, не континуум.
**Вердикт:** *архитектурно ценно сейчас (детерминированная настройка + L0/L1 + параметризация — низкий риск, высокая польза); L3 — честный задел на будущее, безопасный благодаря floor + fallback.* Ровно research-aligned способ сделать «risky» режим ответственно: консилиум-жюри с порогом (§4 кластер 2) + floor/reversibility/phase-gating (§4 кластер 3 + OWASP).

## 6. Anti-over-engineering — сводная памятка (что ресёрч велит НЕ делать)

| Соблазн | Почему ловушка | Рельса |
|---|---|---|
| Открытый completeness-loop «до идеала» | плато к раунду 2-3, риск деградации (Huang et al.) | cap + judge-τ + Δ<ε; граница = DoR, не «идеал» |
| Само-оценка как стоп-сигнал | без внешнего сигнала само-коррекция не работает | внешний oracle/judge/info-gain |
| Консилиум-дебаты на каждое решение | ~15× токенов, groupthink, не бьёт single-CoT на связанном | жюри+гетерогенность; cost-gate; только high-stakes |
| Консилиум авто-решает за человека | против анти-автономного зерна + §4 кластер 2 | информирует гейт; авто только обратимое/дешёвое |
| End-to-end unattended идея→прод | 0.9⁷≈48%, error-compounding, meltdown | гейты на границах фаз; автономия внутри шага |
| Спекулятивный зоопарк персон | токены + groupthink при гомогенности | старт с 3, гетерогенные priors, расширять по пробелам |
| Ручная материализация continuation | хрупко (урок RUN 01) | durable Workflow-скелет; идемпотентные шаги |
| «Полная автономия» = деплой на прод без спроса | необратимый ущерб (OWASP high-privilege) | **floor non-crossable** + reversibility-first |
| Континуум уровней автономности | недетерминизм, неотлаживаемо | фиксированный **enum** + pure-function resolver |
| Консилиум-гейт слепо auto-proceed | падение в прод на низкой уверенности | confidence-порог τ + **human fallback** |
| Диспозиция гейта решается LLM-суждением | дрейф, невоспроизводимость | диспозиция = **код**; LLM только в *содержании* вердикта |
