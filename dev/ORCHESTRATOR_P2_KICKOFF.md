## 1. Назначение и граница

**P2 = процесс поддержки решения на нерешённой архитектурной развилке.** Он НЕ выбирает
архитектуру автоматом — он производит **scored recommendation-пакет для владельца**.

**Почему рекомендация, а не авто-решение (ключевое проектное решение):**
- Вся система маршрутизирует арх-решения владельцу (FB-LR-07 / `remediation-guard`); `PA-042`
  только что сделал ровно это. P2 не должен ломать эту конвенцию — иначе он станет тем самым
  «односторонним резолвом», который гейты запрещают.
- Vision-ресёрч (Epic D): консилиум **проигрывает на одном связанном решении** (~15× дороже +
  groupthink) — **кроме** случая гетерогенного жюри, которое **готовит** решение, а не принимает
  его за пользователя. «100% покрытие пути с гейтами на границах», не «100% автономии».

Итог границы: **P2 сжимает развилку до решаемого вида** (варианты × линзы × риски × рекомендация +
где линзы расходятся), владелец **ратифицирует** → DEC → правка спеков. Автономия P2 — в
*качестве подготовки*, послушание — в *том, что финальный выбор за владельцем*
([[project_autonomy_obedience_balance]]).

**Out of scope P2 (v1):** сам не правит `design.md`/`tech.md`/`tasks.md`; не пишет DEC за
владельца (готовит *черновик* DEC); не исполняет выбранную архитектуру (это P3/P5).

## 2. Контракт триггера (как развилка попадает на вход)

Две двери, одна схема входа:

1. **Из гейта (основной путь):** P6/P4 через `remediation-guard` эскалируют
   `class=cross-spec-conflict` в PA (`PA-040/041/042` — живой пример). Такой PA — **готовый вход
   P2**: у него уже есть перечисленные варианты (a/b/c), маршрут и ссылки на конфликтующие спеки.
   → P2 **потребляет cross-spec-conflict PA**, а не изобретает развилку.
2. **Вручную:** `/orchestrator:run decide-architecture-foundation --fork <PA-NNN | ref>` — когда
   владелец сам хочет прогнать развилку через консилиум.

**Что P2 НЕ делает:** не детектит развилки проактивно (это работа гейтов); нет «сканера
архитектуры». Развилка приходит объявленной.

---

## 3. Схема входа (ForkBrief)

```
ForkBrief {
  fork_id            // PA-NNN или ad-hoc ref
  statement          // в чём развилка (1-2 предложения)
  options[]          // ≥2 взаимоисключающих варианта: { id, summary, mutates[] }
                     //   mutates[] = какие sanctioned-артефакты трогает каждый вариант
  constraints        // выжимка из .product (VP/NFR/product_class) + steering/design pins
  affected_specs[]   // .kiro/specs/*, design.md §, steering — с строками-якорями
}
```

Если вход — cross-spec-conflict PA, `options[]`/`affected_specs[]` **лифтятся из PA** (он их уже
несёт), не собираются заново.

## 4. Движок — жюри ×3 (RA-1 `architecture-consilium`)

`parallel()` из 3 архитекторов, каждый со своим **prior** (гетерогенность = условие, при котором
консилиум вообще работает, Vision D):

| Prior | Линза | Оптимизирует |
|---|---|---|
| **velocity** | скорость доставки / простота / time-to-feedback | быстрее к работающему |
| **fidelity** | верность спекам/steering/design-мандатам | меньше дрейфа от задокументированного |
| **integrity** | целостность рантайма / корректность / отсутствие dead-seam | система реально работает end-to-end |

**Жюри, не дебаты:** без кросс-тока, без раунда консенсуса. Каждый видит один ForkBrief и голосует
независимо. (Дебаты → groupthink + 15× стоимость на связном решении.)

**Verdict-схема каждого архитектора (structured, детерминизирует синтез):**
```
ArchVerdict {
  prior                         // velocity | fidelity | integrity
  scores: { <option_id>: 0..5 } // балл каждому варианту с точки зрения своей линзы
  recommended_option            // топ по своей линзе
  risks_of_recommendation[]     // чем платим, выбрав его рекомендацию (риски СВОЕГО prior)
  blocking_concerns[]           // что делает вариант неприемлемым (если есть)
}
```

---

## 5. Синтез RA-0 (детерминированный, не «thinking»)

RA-0 (orchestrator-controller) агрегирует 3 вердикта **правилом, а не суждением**:

- **Свёртка баллов:** по каждому варианту — вектор (velocity, fidelity, integrity) + агрегат
  (сумма/мин — TBD §9).
- **Консенсус vs раскол:** если все три prior'а сходятся на одном варианте **и** ни у кого нет
  `blocking_concerns` → «сильная рекомендация» (владельцу — почти формальность).
- **Раскол — это и есть решение владельца:** где линзы расходятся (velocity тянет к a,
  integrity — к b), P2 **не форсит консенсус** — он **выпячивает trade-off** как то, что
  владелец обязан взвесить. Это не баг синтеза, а его продукт.
- **Veto:** любой `blocking_concern` понижает вариант ниже рекомендации (нельзя рекомендовать
  вариант, который один из prior'ов считает неприемлемым) — но фиксируется, не скрывается.

## 6. Выход (RecommendationPackage)

```
RecommendationPackage {
  fork_id
  matrix              // option × prior scores (читаемая таблица)
  recommendation      // { option_id, strength: strong|split, rationale }
  the_real_tradeoff   // если split — что именно владелец взвешивает
  dec_draft           // ЧЕРНОВИК DEC-<MODULE>-NNNN (владелец ратифицирует/правит)
  applies_to[]        // какие спеки/доки/таски меняет выбранный вариант (для последующей правки)
}
```

Доставка: пишется в исходный PA (`**Resolution (proposed):**` блок) + `dec_draft` рядом. Владелец
выбирает → флипает PA `done` + коммитит DEC + правит спеки (или заказывает P3/P5 на реализацию).
**P2 сам PA не закрывает и спеки не трогает.**

## 9. Открытые вопросы (решить в дизайне/сборке)

1. **Правило свёртки баллов** — сумма vs мин (worst-prior) vs weighted? Мин консервативнее
   (integrity-veto естественно доминирует); сумма ровнее. → предлагаю **worst-of по blocking +
   сумма по scores для ранга** (зеркало `remediation-guard` worst-of-логики).
2. **Синтез в коде или в промпте RA-0?** Детерминизм говорит «в коде» (`consilium-synth.cjs`), но
   «выпятить настоящий trade-off» — семантическая задача. → предлагаю **гибрид**: код считает
   матрицу+ранг+veto (детерминир.), промпт RA-0 формулирует `the_real_tradeoff`+`rationale`.
3. **Ровно 3 prior'а или конфигурируемо?** v1 — фиксированные 3 (velocity/fidelity/integrity, как
   в RA-1). Расширяемость (напр. cost, security) — post-v1.
4. **Триггер только PA или ещё авто-предложение из гейта?** v1 — P2 запускается по требованию
   (manual или «прогони этот PA»); авто-запуск P2 из P6 при cross-spec-conflict — post-v1 (сначала
   убедимся, что синтез автоматизируем).
