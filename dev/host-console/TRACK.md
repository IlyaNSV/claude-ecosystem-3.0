## 0. Хартия — интенция владельца, дословно (2026-07-15; НЕ пересказывать, цитировать)

> «Хост-пульт управления продуктовым проектом через экосистему, по сути тонкая прослойка,
> которая ловит мои интенции и перекладывает на исполнение процессов в продуктовом проекте
> на VM, следит за сессией, отвечает на вопросы экосистемы в VM продуктовых сессиях, следит
> за логом выполнения, отклонениями, фиксирует сильные и слабые стороны. Пульт должен
> поддерживать управление сразу несколькими, а то и десятком продуктовых проектов. По сути
> ему я буду делегировать управление экосистемой и надзор за её работой. Это своего рода
> global loop управляющий claude сессиями на продукте в VM. Нужно очень хорошо подумать
> над архитектурой этого решения.»

Декомпозиция на 5 ролей: intent intake · dispatch · session supervision + Q&A в мандате ·
observability (лог/отклонения/уроки) · fleet (до десятка проектов).

## 1. Решение о консолидации (владелец, 2026-07-15 — DEC-DEV-0216)

> «Проект кондуктора по сути сводится к текущему global loop, поэтому давай сведем всё это
> к единому проекту и соберем цельную картинку без размазывания по разным проектам.»

**Кондуктор v0 ≡ хост-пульт ≡ global loop — один проект.** Дом — репо `factory-conductor`
(имя и live-валидированный runtime `factory.cjs` уже там). Цельная картинка —
`CONDUCTOR.md` в том репо. Прочие решения владельца той же датой (все по рекомендациям,
rationale — журнал DEC-DEV-0216):

- **Мандат Q&A v0:** auto только `mechanical-ack` + `gate-approve(dev, обратимое)`;
  остальное — эскалация; расширение — только по ledger-данным H2.
- **Побудка Brain:** cron-хартбит (H0/H1) → hook-relay push (с H2). Вечной сессии нет.
- **Q&A-канал:** приоритет файловым (PA/fabric-события); tmux-ввод — только TUI-native
  вопросы под ghost-протоколом.
- **Топология:** один VM с полосами; пересмотр на H3 по данным ledger.

## 2. Зона экосистемы — что фабрика отдаёт пульту (интерфейс, не новые API)

Ecosystem 3.0 — первая фабрика под Кондуктором. Каналы уже существуют и остаются
файловыми/CLI (совместимы с hook-relay каноном; пульт — клиент, не владелец):

| Канал | Механизм | SSOT |
|---|---|---|
| «Где линия» | `fabric-engine.cjs status` (инстансы, состояния) + `state.json`/`events.ndjson` | `dev/process-fabric/CONCEPT.md` |
| Очередь владельца | owner-queue приоритизированная (floor > conflict > gate > review; soft-cap) | там же §4.5 |
| Вопросы/парковки | PA-записи с маркерами `fabric-instance/state/resume-event`; `pa-scan --tick` | `commands/ecosystem/pending-actions.md`, `run.md` |
| Границы автономии | autonomy-policy `resolve(...)`; floor `{prod_deploy, destructive, spend_money, provision_real_secret}` непробиваем | `dev/AUTONOMY_POLICY_F1_CONTRACT.md` |
| Журнал прогонов | run-ledger `runs/ledger.ndjson` (start/finish брекеты) | `commands/orchestrator/run.md` |
| DoD-предикат релиза | отчёт `/product:release-dod` (R1+) — пульт читает в sweep | `dev/release-dod/TRACK.md` |

Жёсткие законы на стороне экосистемы, которые пульт обязан уважать (SSOT формулировок —
`CONDUCTOR.md` §Инварианты): `evt:owner.*` эмитит только владелец (ANOM-OD7-2);
субстрат READ-ONLY для внешнего актора (уроки 54-56); floor непробиваем.

## 3. Очередь трека

1. **H0-спайк** (наблюдатель, read-only sweep) — пре-регистрация и прогон ведутся в
   `factory-conductor` (roadmap — `CONDUCTOR.md`); экосистемная сторона H0 — ничего не

2. Экосистемные работы появляются по фазам пульта: R1-отчёт DoD как вход sweep'а (трек
   release-dod) · hook-relay событий (пререквизит H2 — оценить, что добавить в manifest
   хуков) · при H3 — согласование WIP-лимитов (`limits.json`) с fleet-реестром пульта.
3. Изменения этого файла — только при изменении интерфейса §2 или новых решениях владельца.
