## 0. Хартия — интенция владельца, дословно (2026-07-15; НЕ пересказывать, цитировать)

> «Хост-пульт управления продуктовым проектом через экосистему, по сути тонкая прослойка,
> которая ловит мои интенции и перекладывает на исполнение процессов в продуктовом проекте
> на VM, следит за сессией, отвечает на вопросы экосистемы в VM продуктовых сессиях, следит
> за логом выполнения, отклонениями, фиксирует сильные и слабые стороны. Пульт должен
> поддерживать управление сразу несколькими, а то и десятком продуктовых проектов. По сути
> ему я буду делегировать управление экосистемой и надзор за её работой. Это своего рода
> global loop управляющий claude сессиями на продукте в VM. Нужно очень хорошо подумать
> над архитектурой этого решения.»

Декомпозиция интенции на 5 ролей пульта:
1. **Intent intake** — ловит интенции владельца → раскладывает в процессы конкретного проекта.
2. **Dispatch** — запускает/останавливает Claude-сессии и процессы на VM.
3. **Session supervision + Q&A** — следит за сессией; отвечает на вопросы экосистемы **в пределах мандата**; вне мандата — эскалация владельцу.
4. **Observability** — лог выполнения, deviation watch, фиксация сильных/слабых сторон.
5. **Fleet** — до десятка проектов: реестр, лимиты, планировщик владельческого внимания.

## 1. Главный архитектурный факт (recon 2026-07-15): это — Кондуктор v0

Дизайн meta-system (`META_SYSTEM_DESIGN.md` §7) уже описывает ровно эту прослойку:

> «Кондуктор v0 (эволюция существующих активов, не новый продукт). Собирается из:
> Factory Conductor MVP (tmux-полосы, spawn/harvest) + Process Fabric (charters,

> 1. Intent-триаж… 2. Мост в трекер (ЛК v1)…»

и топологию этажей (анти-дублирование): *Dynamic Workflows оркеструет внутри сессии;
Кондуктор — над сессиями (полосы, VM, проекты, очередь владельца); ЛК — над Кондуктором.*

**Следствие для трека:** мы НЕ проектируем новый пульт с нуля и НЕ строим конкурента
собственной мете. Трек = реализация Кондуктора v0, где Ecosystem 3.0 — первая фабрика

## 3. Архитектура — 6 плоскостей (thin-waist, событийная)

```
ВЛАДЕЛЕЦ ──интенция──▶ [Brain: пульт-сессия на хосте, событийные пробуждения]
                          │  intent-триаж (intent.schema) → WorkOrder → выбор проекта/полосы
                          ▼
              [Policy: мандат = generalization autonomy-policy]
              auto-ответ │ consilium │ эскалация владельцу │ block
                          ▼
   [Dispatch: factory.cjs] ──spawn/send──▶ VM: tmux cf-<lane> × N проектов
                          ▲                    │ hooks сессии
   [Event plane: hook-relay → events.jsonl]◀──┘ (фаза 2 фабрики — пререквизит)
                          │
   [State plane: fleet-реестр + per-project fabric state/owner-queue + run-ledger]
                          │
   [Ledger: сводка/отклонения/уроки → feedback-outbox → Дежурка (когда разморозят)]
```

1. **Substrate** — VM + tmux + worktrees + `factory.cjs`. Существует. Пульт его НЕ мутирует
   (см. инвариант I-1).
2. **Event plane** — hook-relay, не pane-peek (канон меты): хуки сессий пишут атомарные
   event-файлы / `events.jsonl`; пульт читает файлы, не скребёт экран. Пререквизит = фаза 2
   factory-conductor (уже спроектирована там).
3. **State plane** — пульт БЕЗ собственной «памяти в голове»: каждый проход пересобирает
   картину из машинного состояния (fleet-реестр — новый файл; per-project `fabric status`,
   owner-queue, run-ledger — существующие). Recovery-канон: после разрыва — верифицируй, не
   вспоминай.
4. **Policy plane (мандат)** — обобщение autonomy-policy на класс «ответы на вопросы
   сессий»: `resolve(question_class, risk_tier, env) → auto-answer | consilium | escalate |
   block`. Floor наследуется как есть и непробиваем. Классы вопросов v0:
   - `mechanical-ack` (trust-диалог worktree, «continue?», подтверждение чтения) → auto;
   - `gate-approve` обратимого класса на dev-контуре → auto в мандате, лог обязателен;
   - `fork-decision` (ITP-меню, выбор варианта) → только по пре-зарегистрированным
     дефолтам проекта, иначе эскалация;
   - `owner-class` (merge/релиз/prod/деньги/secrets — floor; `evt:owner.*`) → **всегда
     эскалация; пульт НИКОГДА не эмитит owner-события** (I-2).
5. **Brain** — пульт-сессия Claude на хосте. **Не вечная сессия**, а событийные пробуждения
   (cron-хартбит / сигнал hook-relay / ручной вызов): проснулся → sweep флота → ответил в
   мандате / эскалировал / записал → уснул. Каждый sweep — fresh context от машинного
   состояния (инвариант меты №5). Обоснование: вечная TUI-сессия = ресурсная бомба §4.5 +
   деградация контекста + расход лимитов (negative exemplar: сессия 100+ агентов).
6. **UI** — не строим (make-or-integrate меты: «UI не строить»). Ledger пульта — готовый
   фид для размороженной «Дежурки» (read-only трей) и/или ЛК-моста (M3 меты).

## 4. Инварианты (решены уроками кампаний — НЕ пересматривать без новых данных)

- **I-1. Субстрат READ-ONLY для пульта** (урок DEC-DEV-0211/0213, «перечисли акторов»):
  пульт диспетчеризует процессы, но не «чинит» VM/сервисы/окружение сессий; деградация
  субстрата — это данные для вердикта и эскалации, не препятствие к устранению.
- **I-2. `evt:owner.*` — только владелец** (ANOM-OD7-2 guard, DEC-DEV-0174): делегирование
  надзора ≠ делегирование владения. Пульт не проецирует решения владельца, которые «просто
  видел».
- **I-3. Ghost-защита ввода** (8+ случаев ghost-текста в TUI за кампанию, включая опасные
  директивы — `dev/plans/PROD_READINESS_CAMPAIGN.md` журнал): любой ввод в чужую сессию —
  `C-u` → base64-файл → capture-верификация инпута → Enter. Слепой Enter запрещён.
- **I-4. Файл, не проза** (инвариант меты №4/№6): каждый ответ/решение пульта
  материализуется записью (ledger/PA/event) ДО отправки в сессию — иначе его не существует.
- **I-5. Ресурсы**: после harvest сессии гасятся (транскрипты переживают kill-session);
  2-3 полосы на VM — sweet spot; `DISABLE_AUTOUPDATER=1`; WIP-лимиты (`limits.json`
  фабрики) и owner-queue soft-cap — пульт им подчиняется, а не обходит.
- **I-6. Evidence из транскриптов**, не из tmux capture (alternate-screen ограничение).
- **I-7. Владельческое внимание — бюджетируемый ресурс** (инвариант меты №8): эскалации
  батчируются в единую приоритизированную очередь, а не капают поштучно.

## 5. Фазировка (cuttable scope — default; каждая фаза самостоятельно полезна)

| Фаза | Содержание | Что валидирует | Gate |
|---|---|---|---|
| **H0 — Наблюдатель** (спайк) | Read-only sweep: пульт-сессия по cron/вручную проходит N проектов (ssh → fabric status, owner-queue, run-ledger, транскрипт-хвосты) → одна сводка + список эскалаций | Плоскости Event/State без риска; ценность «следит за логом и отклонениями» сразу | без пререквизитов |
| **H1 — Диспетчер** | Intent → запуск процесса на 1 проекте (`factory spawn/send` с брифом), брекет в ledger, harvest; каждый dispatch ратифицирует владелец | Intent-триаж + Dispatch | H0 прожит |
| **H2 — Q&A в мандате** | Автоответы классов `mechanical-ack`/`gate-approve(dev)`; ghost-протокол I-3; всё в ledger; owner-class — эскалация | Policy plane | **gated на events.jsonl** (фаза 2 фабрики) — отвечать по pane-peek запрещено каноном |
| **H3 — Флот** | Мульти-проектный реестр, WIP/внимание-бюджет, единая очередь эскалаций поверх проектов | Fleet, «десяток проектов» | H1+H2 на 2 проектах |
| **H4 — Контур обучения** | Deviation watch формализован; strengths/weaknesses ledger → feedback-outbox → доработки экосистемы | Замыкание global loop | live-опыт H3 |

Связка с треком Release DoD: сводка H0 включает состояние DoD-предиката по каждому
активному RL; H3-планировщик приоритизирует проекты, у которых DoD-цикл ждёт владельца.

## 6. Экономика и риски (честно)

- **Стоимость**: пульт — LLM-сессия; каждый sweep жжёт токены. Митигируется: событийные
  пробуждения вместо вечной сессии, батчинг эскалаций, детерминированная механика (`factory
  status --json`, `fabric status`) без LLM там, где можно. Замер $/sweep — обязательная
  часть H0 (инвариант меты №10).
- **Риск делегирования**: авто-ответ не того класса = тихое решение за владельца. Митигация:
  мандат = код (resolve), floor непробиваем, I-2/I-4, эскалация по умолчанию при сомнении.
- **Риск рассинхрона с метой**: если Кондуктор строится и здесь, и в meta-system — два
  конкурента. Митигация: §7.1 — одна реализация, ратифицированная владельцем.
- **VM-ресурсы**: 8 ГБ/12 vCPU текущей VM держат 2-3 полосы; «десяток проектов» ≠ десяток
  одновременных полос — планировщик внимания важнее параллелизма (H3).

## 7. Вопросы владельцу (до H0)

1. **Место кода и идентичность** — пульт строим как Кондуктор v0 по дизайну меты:
   runtime-код в репо `factory-conductor` (эволюция, фаза 2 там уже спроектирована),
   контракты из `meta-system/contracts/`, экосистема отдаёт машинные каналы (fabric/PA/
   autonomy) без изменений (рекоменд.) — или отдельный модуль в ecosystem-3.0 / код в
   meta-system? Трек-канон (журнал, гейты, паттерны) в любом случае ведём здесь.
2. **Мандат v0** — утвердить классы §3.4: auto только `mechanical-ack` +
   `gate-approve(dev, обратимое)`; всё остальное — эскалация. Расширение мандата — только
   по накопленному ledger H2 (данные, не доверие).
3. **Канал побудки Brain** — cron-хартбит на хосте (рекоменд. для H0) vs hook-relay push
   vs ручной запуск. (Вечную сессию не предлагаю — §3.5.)
4. **Q&A-канал** — приоритет файловым каналам (PA/fabric-события), tmux-ввод только для
   вопросов, живущих исключительно в TUI (trust-диалог), с I-3 — ратифицировать?
5. **Топология VM** — один VM с полосами (сейчас) или VM-на-проект при росте флота?
   Влияет на H3; для H0/H1 достаточно текущей.
