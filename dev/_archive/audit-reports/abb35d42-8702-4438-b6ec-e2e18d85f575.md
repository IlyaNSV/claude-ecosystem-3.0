## Process catalog findings

### 🔴 Blocking

**[F] Handoff regeneration never persisted; destructive workaround on wrong path
(worktree/main-repo split).** — `.product/handoffs/FM-001-handoff.md`

The skill composed a full v2 handoff, but every persistence attempt failed and the
recovery workaround degraded the artifact instead of writing it.

- Three identical Write failures (lines 147, 155, 165):
  > `<tool_use_error>File has not been read yet. Read it first before writing to it.</tool_use_error>`
- Destructive workaround executed before any successful write (line 153):
  > Bash `rm ".product/handoffs/FM-001-handoff.md" && echo "removed"` → `removed`
  > Bash `printf 'placeholder for regeneration\n' > ".product/handoffs/FM-001-handoff.md"` → `created placeholder`
- cwd diagnosis (line 163, verbatim) confirms the path split — the placeholder
  landed in the worktree while the real handoff sits in the main repo:
  > `PWD=/c/Users/pw201/WebstormProjects/my-first-test/.claude/worktrees/purring-popping-prism`
  > `---worktree abs--- … -rw-r--r-- … 29 Jun 1 13:18 FM-001-handoff.md`
  > `---main repo abs--- … -rw-r--r-- … 210753 May 15 00:10 FM-001-handoff.md`

Root cause (hypothesis, high confidence in the symptom): the Bash tool runs with
`PWD` = the git worktree, so `rm`/`printf` on the relative path hit the worktree
copy; the `Write`/`Read` tools were handed the **main-repo** absolute path
(`C:\Users\pw201\WebstormProjects\my-first-test\.product\handoffs\FM-001-handoff.md`,
no `worktrees\…` segment, 210 KB), which cannot be fully `Read` (exceeds the token
cap), so the read-before-write guard could never be satisfied. Net: v2 was never
written anywhere.

Outcome / risk:
- The stale v1 the user wanted refreshed is **still stale** — main-repo copy
  unchanged (210753 B, May 15). No regression to the canonical file.
- The worktree now holds a **29-byte placeholder stub**. This is a merge landmine:
  if the `purring-popping-prism` branch is merged/committed, the stub would clobber
  the 210 KB handoff.

Recommended action: do NOT `rm` a handoff before a successful regeneration write —
overwrite in place (the spec treats regeneration as full overwrite, git holds
history; handoff-spec §10). Harden the handoff command/skill for git-worktree
execution: resolve paths against the actual `cwd`/worktree (not a hard-coded
main-repo absolute path), and read-before-write the existing handoff (or stream the
write via Bash) so the guard is satisfiable on large files. Clean up the worktree
29-byte stub or re-run the regeneration from the project root before any merge.
