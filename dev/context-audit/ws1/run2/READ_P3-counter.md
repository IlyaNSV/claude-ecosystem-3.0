# READ_P3-counter — контр-тезис: длинный контекст ПОМОГАЕТ

**Прогон:** WS1 / run2 · **Приоритет:** P3-counter · **Дата чтения:** 2026-07-12
**Мандат:** найти свидетельства ПРОТИВ нашего нарратива «резать контекст». Взвешивание —
по `decision-delta × (1 − indirectness)`, НЕ по тиру S-лестницы.

---

## 0. TL;DR — что реально изменилось после чтения

**Главный вывод: наша ось спора («сколько резать») почти наверняка НЕ ТА ОСЬ.**

Существуют **две работы, измеряющие ровно наш артефакт** (CLAUDE.md / AGENTS.md для кодинг-агентов,
одна — прямо в Claude Code CLI). Обе **не находят ДЛИНУ управляющей переменной**:

- **ETH Zurich / LogicStar (arXiv 2602.11988):** вредит не объём, а **избыточность**. LLM-сгенерированный
  контекст-файл (который пересказывает то, что и так лежит в репо) даёт **−2%** успеха; **написанный
  человеком** — **+4%**. Уберите из репо всю остальную документацию — и тот же LLM-файл становится
  **+2.7%** и обгоняет человеческий. Авторы прямо пишут: длину как отдельную переменную они не
  анализировали, ключ — **новизна содержимого, а не объём**.
- **McMillan (arXiv 2605.10039), 1 650 сессий Claude Code CLI:** **размер файла — подтверждённый
  байесовский НОЛЬ** (BF10 = 0.05–0.10, т.е. свидетельство В ПОЛЬЗУ отсутствия эффекта, а не просто
  «не нашли»). Позиция инструкции — ноль. Архитектура файла — ноль. **Противоречия между соседними
  файлами — тоже подтверждённый ноль.** Единственный устойчивый эффект — **внутрисессионный распад**:
  −5.6% шансов соблюдения на каждую следующую сгенерированную функцию (OR = 0.944).

Плюс: **счёт правил в десятках — не связывающее ограничение.** IFScale (20 моделей, вкл. Claude
Opus/Sonnet 4): фронтир держит **~98% при 150 инструкциях**, o3-high — **97.8% при 250**; обвал
начинается к **500**. У нас правил — десятки.

И: **большой контекст покупает способности, которых мало-контекстный режим не даёт вовсе** —
many-shot ICL перебивает pretraining-биасы (flipped labels), чего few-shot не может в принципе
(Agarwal, NeurIPS 2024 spotlight).

**Следствие для аудита:** рекомендация «режь CLAUDE.md, он длинный» **не подпёрта** прямыми
свидетельствами. Подпёрта другая: **«режь то, что агент и так достанет через grep/glob за один шаг;
держи то, что недостижимо иначе»**. И — неожиданно — **главный рычаг деградации лежит не в файле,
а во времени сессии** → лечится ре-инжекцией (хуки!), а не укорачиванием файла.

## 1. Прямые свидетельства по НАШЕМУ артефакту (indirectness = none)

### 1.1 ⭐ ETH Zurich / LogicStar — «Evaluating AGENTS.md: Are Repository-Level Context Files Helpful for Coding Agents?»

**Источник:** arXiv:2602.11988v1 (12 Feb 2026). Gloaguen, Mündler, Müller, Raychev, Vechev (ETH Zurich /
LogicStar.ai). **Препринт** (тир S2; рецензирование не подтверждено — см. §5).

**Дизайн:**
- **AGENTbench** — новый сьют: **138 Python-задач из 12 «нишевых» репозиториев**, где есть
  *настоящие, написанные разработчиками* контекст-файлы.
- **SWE-bench Lite** — 300 задач из популярных репо (там своих контекст-файлов нет).
- **4 фронтир-связки:** **Claude Code + Sonnet-4.5**, Codex + GPT-5.2, Codex + GPT-5.1-mini,
  Qwen Code + Qwen3-30b-coder.
- **3 условия:** без файла / LLM-сгенерированный файл / файл разработчика.

**Числа (успех решения задачи):**

| Условие | SWE-bench Lite | AGENTbench |
|---|---|---|
| LLM-сгенерированный контекст-файл | **−0.5%** | **−2%** |
| Файл, написанный разработчиком | н/д (нет файлов) | **+4%** |

**Цена (это важно — файл не бесплатен):**
- Инференс: **+20–23%** (LLM-файл), до **+19%** (человеческий файл).
- Шаги: **+2.45** (SWE-bench) / **+3.92** (AGENTbench) для LLM-файла; **+3.34** в среднем для
  человеческого.
- Reasoning-токены: **+22%** (GPT-5.2 с LLM-файлом), **+14%** (GPT-5.1-mini), **+20%** (GPT-5.2 с
  человеческим файлом).

**🔑 Ключевой эксперимент (гипотеза избыточности).** Авторы **удалили из репозиториев всю прочую
документацию** — все `.md`, примеры кода, каталоги `/docs/` — и заново прогнали агентов с
LLM-сгенерированным контекст-файлом как **единственным** источником документации.
**Результат: тот же файл дал +2.7% и обогнал человеческий.**

> Интерпретация авторов: LLM-файлы в основном **дублируют** уже обнаружимую в репо документацию.
> Когда документация исчезает, синтезированный файл становится ценным. Человеческие файлы помогают
> потому, что несут **новое** знание сверх README.

**Поведенческий разбор «почему хуже»:** контекст-файлы повышают частоту прогонов тестов, grep,
чтения/записи файлов; упомянутые в файле инструменты (`uv`, кастомные билд-системы) используются
в **1.6–2.5×** чаще. Агенты **послушно следуют** файлу — но начинают **шире исследовать кодовую базу
без пропорционального роста точности**: «exploring more, learning less».

**Что авторы называют вредным:** избыточный пересказ существующей документации; предписания
инструментов, которые гонят агента исследовать, не приближая к решению; обзорные разделы «структура
репозитория» с перечислением каталогов.
**Что полезным:** новая, необнаружимая иначе информация; минимальные конкретные требования
(«для зависимостей используй X»).

**Прямая цитата-суть (по извлечению):** *«Context file length was not explicitly analyzed as an
independent variable; instead, the authors emphasize content novelty rather than volume.»*

**Рекомендация авторов:** **не класть LLM-сгенерированные контекст-файлы** (вопреки официальным
гайдам вендоров агентов); человеческие — держать **минимальными и сфокусированными на
domain-specific требованиях, которые не обнаружить самому**.

**Ограничения (их):** только Python; только issue-resolution/feature-addition; нишевые репо могут
систематически отличаться от больших популярных; как генерировать хороший контекст-файл автоматически —
открытая проблема.

**decision-delta для нас: МАКСИМАЛЬНЫЙ.** Это единственная работа, которая измеряет наш класс файла
на нашем агенте (Claude Code) — и она переопределяет критерий реза с *объёма* на *новизну*.

### 1.2 ⭐ McMillan — «Instruction Adherence in Coding Agent Configuration Files: A Factorial Study of Four File-Structure Variables»

**Источник:** arXiv:2605.10039 (11 May 2026), Damon McMillan. **Одноавторский препринт**, не
рецензирован, **НЕ преregistered** (главный эффект найден на этапе анализа — автор честно это пишет).

**Дизайн:** факторный эксперимент, **1 650 сессий Claude Code CLI**, **16 050 наблюдений на уровне
функций**, **2 TypeScript-кодовые базы**, **3 фронтир-модели** (в основном **Sonnet 4.6**; **Opus 4.6**
как cross-model проверка при совпадающей версии CLI; Opus 4.7 — описательно, под CLI-version
конфаундом), **5 кодинг-задач**. Смешанные модели (mixed-effects) + байесовский компаньон.

**4 манипулируемые переменные:** (1) **размер файла**, (2) **позиция инструкции**, (3) **архитектура
файла**, (4) **противоречия в соседних файлах**.

**Метрика:** соблюдение **одной тривиальной целевой аннотации**.

**Результаты — verbatim из абстракта:**

> «None of the four structural variables or three two-way interactions produces a detectable contrast
> after multiple-testing correction. Size and conflict nulls are supported by affirmative-null Bayes
> factors (BF10 between 0.05 and 0.10); position and architecture nulls are failures to reject without
> Bayes-factor support. The largest effect we measured is within-session: each additional function the
> agent generates is associated with approximately 5.6% lower odds of compliance per step (OR = 0.944)
> within the session-length range we tested, though the relationship is non-monotonic rather than a
> constant per-step effect.»

**Разбор силы утверждений (важно — не все нули одинаковы):**

| Переменная | Статус нуля | Сила |
|---|---|---|
| **Размер файла** | **affirmative null**, BF10 0.05–0.10 | 🟢 свидетельство В ПОЛЬЗУ отсутствия эффекта (10–20× в пользу H0) |
| **Противоречия в соседних файлах** | **affirmative null**, BF10 0.05–0.10 | 🟢 то же |
| Позиция инструкции | failure to reject, без байес-поддержки | 🟡 просто «не нашли» |
| Архитектура файла | failure to reject, без байес-поддержки | 🟡 просто «не нашли» |

**Единственный устойчивый эффект — внутрисессионный:** OR = 0.944 на каждую следующую
сгенерированную функцию. Воспроизводится на второй TS-кодовой базе и на Opus 4.6 при совпадающей
конфигурации. Немонотонный.

**decision-delta: ОЧЕНЬ ВЫСОКИЙ, но метрика узкая.** Это ПРЯМО Claude Code. Оно говорит:
- (а) наши споры про **размер / позицию правила / модульность файла / противоречие global↔project
  CLAUDE.md** — по крайней мере для простого соблюдения — **вероятно, спор ни о чём**;
- (б) **реальный враг — глубина сессии**, а не файл.

**⚠️ Честная оговорка (критично, не сглаживать):** метрика — соблюдение **ОДНОЙ тривиальной
аннотации**. Это НЕ тест на «деградирует ли сложное многоправильное поведение под 20k-токенным
CLAUDE.md». **Уровни переменных (что именно значит «большой файл» в токенах) мне извлечь не удалось**
(HTML — 404, PDF — бинарь). Если «большой» = 2k токенов, ноль не говорит ничего про наши объёмы.
→ Это HYPOTHESIS, не RULE.

### 1.3 ⚠️ Lulla et al. — «On the Impact of AGENTS.md Files on the Efficiency of AI Coding Agents» (и ПОЙМАННАЯ ПОДМЕНА)

**Источник:** arXiv:2601.20404v2. Lulla (SMU), Mohsenimofidi (Heidelberg), Galster (Bamberg),
Zhang (KCL), Baltes (Heidelberg), Treude (SMU). Journal Ahead Workshop (JAWs) 2026.

**Дизайн:** OpenAI Codex + gpt-5.2-codex; **124 реальных pull request** из 10 репозиториев;
парный внутризадачный дизайн (каждая задача — дважды, с файлом и без). PR ≤100 LoC, ≤5 файлов.

**Результаты (Wilcoxon signed-rank, p<0.05):**
- **Медианное wall-clock время: −28.64%** при наличии AGENTS.md
- **Медианные output-токены: −16.58%**
- **Средние output-токены: −20.08%**

Вывод авторов: наличие корневого AGENTS.md **ускоряет и удешевляет** работу на реальных PR.

**🚨 МЕТОДОЛОГИЧЕСКАЯ ЛОВУШКА, В КОТОРУЮ Я ЧУТЬ НЕ ПОПАЛ.** Поисковая выдача **склеила две разные
работы**: она приписала статье Lulla et al. числа из ETH-работы («снижает успех на 3%, раздувает
стоимость на 20%»). Это **неверно** — я проверил обе по первоисточникам. Это **разные статьи с
частично ПРОТИВОПОЛОЖНЫМИ результатами по цене**:

| | ETH (2602.11988) | Lulla (2601.20404) |
|---|---|---|
| Агент | Claude Code/Sonnet-4.5, Codex, Qwen | Codex/gpt-5.2-codex |
| Задачи | бенчмарки (AGENTbench 138, SWE-bench Lite 300) | 124 реальных PR |
| **Стоимость с файлом** | **+20–23% инференс** | **−16.6…−20% output-токенов, −28.6% времени** |
| Корректность | измерена (±0.5–4%) | **НЕ измерена комплексно** (санити-чек на 50 PR) |

Скорее всего оба правы по-своему: ETH считает **полную стоимость с reasoning-токенами и шагами** на
бенчмарках; Lulla — **output-токены и время** на реальных мелких PR. Но напряжение реально, и

## 2. Контр-тезис в общем виде: где БОЛЬШЕ контекста = ЛУЧШЕ

### 2.1 ⭐ Agarwal et al. — «Many-Shot In-Context Learning» (NeurIPS 2024 **spotlight**)

**Источник:** arXiv:2404.11018. Agarwal, Singh, Zhang, Bohnet, … Larochelle (Google DeepMind).
Тир **S1** (NeurIPS spotlight). Модель: **Gemini 1.5 Pro (1M токенов)**.

**Выигрыши (условия — точные):**

| Задача | Оптимум | Эффект |
|---|---|---|
| MT Bemba (low-resource) | **997 шотов (~85K токенов)** | **+15.3%** vs 1-shot |
| MT Kurdish | 997 шотов | **+4.5%**, SOTA |
| Планирование (Logistics) | ~10+ шотов | **37% → 62%** успеха |
| Big-Bench Hard | ~100 шотов | Reinforced ICL **83%** vs **72%** CoT |
| GPQA | **125 шотов** | ≈ уровень Claude-3 Opus |
| Code verifier (GSM8K) | 128+ шотов | Pass@1 подтянут к Pass@4 |

**🔥 Самое сильное для нас — many-shot ПЕРЕБИВАЕТ pretraining-биасы, чего few-shot не может:**
на flipped-labels и abstract-labels (A/B/C) sentiment few-shot проваливается, но *«as the number of
shots increases, performance on flipped and abstract labels dramatically improves, approaching that of
default labels»*. Механизм — «early ascent»: малое число шотов вытаскивает НЕВЕРНЫЙ навык, и это
уходит только с масштабом.
→ **Это прямой контр-аргумент к «меньше контекста лучше»: есть класс поведений, который покупается
ТОЛЬКО объёмом.** Если мы хотим, чтобы агент делал что-то ПРОТИВ своих дефолтов (а весь наш
CLAUDE.md ровно об этом — ITP-маркеры, «без эмодзи», «без двоеточия перед тулом», MDP-тиры), то
скупой контекст — ровно неправильная стратегия.

**И честная граница ОТТУДА ЖЕ (не монотонно!):**
- **XSum (суммаризация): деградирует после 50 шотов.**
- **MATH: пик ~125 шотов, дальше спад.**
- **GPQA: деградация на 250 шотах.**
- Code verifier: лёгкое ухудшение после некоторого числа шотов.

**Бонус-урок про метрики:** *«NLL trends are not a strong predictor for downstream task performance»* —
на MATH/GPQA успех падал после 125 шотов, **а NLL при этом не рос**. → **Не доверяй прокси-метрикам
качества контекста; меряй задачу.** (Прямо релевантно дизайну нашего модуля добора: «perplexity
контекста» — негодный критерий.)

### 2.2 ⭐ Bertsch et al. — «In-Context Learning with Long-Context Models» (NAACL 2025)

**Источник:** arXiv:2405.00200. Bertsch, Ivgi, Xiao, Alon, Berant, Gormley, Neubig (CMU/Tel Aviv).
Тир **S1**.

**Модели:** Llama2-7b (4k), Llama2-32k, **Llama2-80k**, Mistral-7b-v0.2 (32k).
**Датасеты (по числу меток):** TREC (6), TREC-fine (50), NLU (68), **Banking-77 (77)**, **Clinic-150 (151)**.

**Числа:**
- **10 → 1000 демонстраций: до +50.8 пункта, в среднем +36.8 пункта** по 5 датасетам (Llama2-80k).
- Рост **монотонный почти на каждом датасете** — обвала нет; точка насыщения зависит от датасета
  (TREC насыщается на ~20 примерах; Banking-77 не насыщается даже в 4k-окне).
- Выигрыш концентрируется в **больших пространствах меток**.
- **Преимущество retrieval-отбора примеров тает с масштабом:** Banking-77 — **51.5 пункта на 1-shot →
  4.9 пункта на 1500-shot**; в 2000-shot разрыв **никогда не превышает 5 пунктов**. → на длинном
  контексте **случайный отбор ≈ умный отбор**.
- **Длинно-контекстный ICL МЕНЕЕ чувствителен к перемешиванию входа, чем коротко-контекстный.**
  ← прямой контр-аргумент к интуиции «порядок правил критичен».
- Группировка одинаковых меток подряд — **вредит**.
- Блочное внимание: выигрыш идёт **не** от «совместного кодирования» примеров, а от «attending back
  to similar examples».

### 2.3 SPRIG — «Improving LLM Performance by System Prompt Optimization» (ICLR 2026)

**Источник:** arXiv:2410.14826. Zhang, Ergen, Logeswaran, Lee, Jurgens.

- **47 типов задач / 42 бенчмарка / 7 категорий** (reasoning, math, social, commonsense, faithfulness,
  knowledge, language-understanding). Модели: Llama3.1-8B, Mistral-Nemo, Qwen2.5-7B; генерализация —
  на Llama3.1-70B, Mistral-Large, Qwen2.5-72B.
- **Оптимизированный СИСТЕМНЫЙ промпт: ~+10% к среднему баллу** vs неоптимизированный.
- Оптимизированный **системный** промпт **работает наравне с промптами, оптимизированными под каждую
  задачу отдельно**; system + task **комплементарны** (ещё +1.6% на больших моделях).
- **Генерализуется** между семействами моделей и размерами (task-промпты — падают, p<0.01) и по
  языкам (значимый рост на 4 из 5 мультиязычных бенчмарков).
- Таксономия компонентов (9 категорий, ~300 штук): good-property (146), role (43), style (22),
  emotion (17), scenario (13), jailbreak (9), safety (16), behavioral (16), CoT (18).
- **🔪 Готовое правило реза:** компоненты типа **«good property»** («ты — эмпатичный ассистент»)
  выбираются оптимизатором **значительно РЕЖЕ, чем случайно**, — при том что это самый популярный
  дефолт у практиков. Ценность несут **CoT- и role-компоненты**.
- Максимальный выигрыш — на math/reasoning; минимальный — на knowledge-задачах.

→ **Контр-тезис:** «системный промпт должен быть минимальным» — неверно. Хорошо собранный системный
слой стоит ~10 пунктов и переносится между моделями. **Но** декоративные «ты — заботливый помощник»
строки — мусор, и это измерено.

### 2.4 Inference Scaling for Long-Context RAG (ICLR 2025)

**Источник:** arXiv:2410.04343. Yue, Zhuang, Bai, Hui, Jagerman, Zeng, Qin, Wang, Wang, Bendersky (Google).
- *«Increasing inference computation leads to nearly linear gains in RAG performance when optimally
  allocated»*; **до +58.9%** к стандартному RAG на бенчмарках.
- Оговорка авторов: простое раздувание контекста **само по себе** не гарантирует улучшения — важна
  **оптимальная аллокация** вычислений/контекста.
→ Умеренный контр-аргумент: длина полезна **при правильном распределении**, а не сама по себе.

### 2.5 Память агентов (только поисковый уровень, первоисточники НЕ читал)

Сигналы, что персистентный контекст **помогает** агентам: SWE-ContextBench (oracle-саммари прошлых
траекторий → **+8 п.п.** resolution); AgeMem — устойчивое превосходство над memory-augmented
бейзлайнами на 5 long-horizon бенчмарках; на TravelPlanner/ALFWorld успех растёт по мере «дозревания»
репозитория памяти. **Это НЕ прочитано в первоисточнике** → только как направление, не как опора.

## 3. Граница: где выигрыш кончается (честные контр-контр-свидетельства)

Мандат требует искать, где мы неправы. Но так же честно — где **контр-тезис** неправ.

### 3.1 IFScale — «How Many Instructions Can LLMs Follow at Once?» (arXiv 2507.11538)

**Дизайн:** плотность инструкций **от 10 до 500**; задача — писать бизнес-отчёт, включив заданные
ключевые слова (из SEC 10-K) без морфологических вариаций. **20 моделей**, включая **claude-opus-4,
claude-sonnet-4, claude-3.7-sonnet, claude-3.5-haiku** (+ reasoning-варианты), gpt-4.1/4o/4.5, o3,
gemini-2.5-pro/flash, deepseek-r1, grok-3, qwen3, llama-4.

**Числа (точность соблюдения):**

| Плотность | Лучшая модель | Точность |
|---|---|---|
| 10 | несколько | ~98–100% |
| 50 | gemini-2.5-pro | **99.6%** |
| 150 | gemini-2.5-pro | **~98%** |
| 250 | o3 (high) | **97.8%** |
| 500 | o3 (high) | **62.8%** |

Худший: gpt-4o **94% → 15.4%** при 500.

**Три паттерна распада:**
1. **Пороговый** — почти идеально до критической плотности, потом обрыв (o3, gemini-2.5-pro, grok-3).
2. **Линейный** — ровное падение с самого начала (**gpt-4.1, claude-3.7-sonnet**).
3. **Экспоненциальный** — быстрый ранний обвал, дно 7–15% (**claude-3.5-haiku**, llama-4-scout).

**Primacy bias:** максимален при **средних** плотностях (150–200) — модель начинает предпочитать
ранние инструкции именно там, где «начинает надрываться»; при 300+ смещение исчезает в пользу
равномерного отказа («точка насыщения инструкциями»).
**Тип ошибок:** при высокой плотности — почти исключительно **пропуски** (llama-4-scout: 34.88:1
omission:modification). Reasoning-модели держат более низкое отношение.
**Латентность:** o3 — **26.3 с (10 инструкций) → 219.6 с (250)**.

**Что это даёт нам:**
- ✅ **Контр:** наш счёт правил (десятки) — **глубоко внутри «почти идеального» режима** фронтира.
  Резать правила из страха «перегрузки» — не подпёрто.
- ⚠️ **Но осторожно:** **claude-3.7-sonnet попал в ЛИНЕЙНЫЙ распад** — у Claude может не быть
  «бесплатного» плато; деградация постепенная с самого начала. Это единственное найденное свидетельство
  в пользу «каждое правило что-то стоит даже на малых числах» — и оно про Claude.

### 3.2 «Curse of instructions» / When Instructions Multiply (arXiv 2509.21051, Findings EMNLP 2025)

**Дизайн:** **ManyIFEval** (текст, до **10** инструкций), **StyleMBPP** (код, до **6**); 10 LLM
(GPT-4o, Claude 3.5 Sonnet, Gemini 1.5 Pro, DeepSeek-R1, …). Метрика — **prompt-level accuracy**
(соблюдены ВСЕ инструкции сразу).

**Числа (ManyIFEval, n=1 → n=10):**
- GPT-4o: **0.94 → 0.21**
- Claude 3.5 Sonnet: **0.95 → 0.48**
- DeepSeek-R1: **0.95 → 0.38**

Родственная работа («Curse of Instructions», OpenReview R6q67CDBCH) формулирует «проклятие»:
success_all ≈ success_single^N. Данная статья уточняет: **строгая мультипликативность НЕ
подтверждается**, логистическая модель описывает лучше → есть **взаимодействия** между инструкциями.
Reasoning-модели заметно лучше (явно трекают инструкции по очереди); митигация — итеративный
self-refinement / instruction-level CoT.

**🧩 Как это мирится с IFScale (важно — кажущееся противоречие):**
IFScale меряет **точность на инструкцию** (~98% каждая при 150). ManyIFEval меряет **«соблюдены ВСЕ
одновременно»** — и это композиция, которая складывается вниз. **Оба верны.**
→ **Дорого не «много правил в файле», а «много правил, обязанных сработать в ОДНОМ ответе».**
Условно-триггерные правила (сработают, когда сработают) — дёшевы. Всегда-активные — компонуются.

**Для нас это конкретно:** посчитать, сколько правил CLAUDE.md обязаны выстрелить **одновременно, в
одном ответе** (ITP-маркер + нет эмодзи + нет двоеточия перед тулом + «Короткий ответ» + …).
**Вот это число — дорогое.** Общий объём файла — нет.

### 3.3 «Context Length Alone Hurts LLM Performance Despite Perfect Retrieval» (Findings EMNLP 2025, arXiv 2510.05381)

**Дизайн:** 5 моделей — Llama-3.1-8B, Mistral-v0.3-7B, **GPT-4o, Claude-3.5-Sonnet, Gemini-2.0**;
задачи — GSM8K (math), MMLU (QA), HumanEval (code), VarSum (синтетика). Длины: 0 / 3.75K / 7.5K / 15K /
26.25K / 30K токенов.

- Падение **13.9% – 85%** — **в пределах заявленных окон моделей**.
- **Бóльшая часть падения происходит уже в пределах ~7K токенов.**
- Держится даже при **маскировании** нерелевантных токенов (модель физически вынуждена смотреть
  только на нужное): **≥7.9%** просадки при 30K маскированных токенов → **виновата сама длина, а не
  отвлечение**.
- Митигация «retrieve-then-solve» (продиктуй улику, потом решай): **до +4%** для GPT-4o на RULER QA.

**Честная деконструкция (это важно — числа не то, чем кажутся):**
- **85%** = **Llama-3.1-8B на синтетическом VarSum при 30K**.
- **13.9%** = **Mistral-7B на MMLU при 7.5K**.
- Т.е. **драматические концы диапазона держат маленькие открытые модели и синтетическая задача.**
  Авторы прямо пишут: *«closed-source models generally exhibit more robustness than the open-source ones»*.
- ⚠️ Пер-модельные ячейки по фронтиру я **не смог извлечь надёжно** (в таблице вылезли дублирующиеся
  значения −41.7% для двух разных задач Claude — явный артефакт парсинга). **Не опираюсь на них.**

**Вывод:** работа реальна и поддерживает наш приор — но **в более слабой форме**, чем поисковая выдача
рекламирует: «длина стоит чего-то», а не «длина убивает фронтир на реальных задачах».

### 3.4 Гайд Anthropic (вендорское, тир S4 — но про НАШ харнесс)

«Effective context engineering for AI agents»:
- **Против обеих крайностей:** плохо и «hardcoding complex, brittle logic», и «vague, high-level
  guidance». Цель — *«the minimal set of information that fully outlines your expected behavior»*, и
  прямо: **«minimal does not necessarily mean short»**. ← Вендор нашего харнесса **против** наивного реза.
- «Right altitude» — золотая середина.
- Про context rot: ссылаются на исследования, **но признают** — это *«performance gradient rather than
  a hard cliff»*, без чисел и порогов.
- Про CLAUDE.md прямо: *«CLAUDE.md files are naively dropped into context up front, while primitives
  like glob and grep allow it to navigate its environment and retrieve files just-in-time»* → предлагают
  **just-in-time** вместо пред-загрузки; гибрид допустим.

### 3.5 ⚠️ Конфликт интересов у источника «context rot»

Самый цитируемый носитель нарратива «длинный контекст гниёт» — **Chroma** (`trychroma.com/research/context-rot`).
**Chroma продаёт векторную БД / RAG-инфраструктуру.** Если длинный контекст плох — это **прямо
поддерживает их продукт**. Это **не опровержение** (их методология как раз аккуратна — они сами
критикуют бенчмарки за смешивание длины входа со сложностью задачи), но это **флаг ангажированности
источника**, который наш приор, возможно, впитал некритично. Аналогично: половина «context
engineering»-контента исходит от вендоров memory/RAG-продуктов.

## 4. Синтез: что это меняет в решении

### 4.1 Что НЕ подпёрто (и должно быть снято или переформулировано)

| Наш приор | Вердикт после чтения |
|---|---|
| «CLAUDE.md длинный → режь по объёму» | ❌ **Не подпёрто.** ETH: длина не была управляющей переменной, ключ — новизна. McMillan: **affirmative-null на размер файла в Claude Code** (BF10 0.05–0.10). |
| «Много правил → перегрузка инструкциями» | ❌ **Не подпёрто на наших числах.** IFScale: ~98% при 150 инструкциях у фронтира. Мы в десятках. |
| «Порядок/позиция правил критичны» | ❌ **Не подпёрто.** McMillan: позиция — ноль. Bertsch: длинный контекст **менее** чувствителен к порядку, чем короткий. (Плюс бриф уже запретил тащить сюда Lost-in-the-Middle.) |
| «Противоречие global↔project CLAUDE.md — дорогая проблема» | ⚠️ **Скорее не подпёрто:** McMillan — affirmative-null на «contradictions in adjacent files». (Метрика узкая → перепроверить у себя.) |
| «Меньше контекста → лучше поведение» | ❌ **Опровергается** там, где нужно перебить дефолт модели: many-shot **перебивает pretraining-биасы**, few-shot — **не может** (Agarwal). Наш CLAUDE.md ровно про «делай ПРОТИВ дефолта». |

### 4.2 Что подпёрто (и должно стать критерием)

1. **🔪 Критерий реза = ИЗБЫТОЧНОСТЬ, а не объём.**
   ETH (прямой эксперимент): файл, дублирующий обнаружимую документацию, **вредит** (−2%); он же
   **помогает** (+2.7%), если дублировать больше нечего. Человеческий (новое знание) — **+4%**.
   → **Правило: удаляй из CLAUDE.md всё, что агент достаёт из репо за 1–2 вызова grep/glob/read.
   Оставляй то, что нигде не лежит: конвенции, «мы пробовали X и это провалилось», решения, грабли.**
   (Заметь: наш CLAUDE.md как раз во многом **ссылочный** — «SSOT = ROADMAP», «детали в checklists/*».
   По ETH это **правильная** архитектура: указатель дёшев, пересказ дорог. А вот блоки, дублирующие
   README/ROADMAP/структуру репо — кандидаты №1 на рез.)

2. **💰 Файл не бесплатен — он обязан окупаться.**
   ETH: **+20–23% инференса, +2.5–3.9 шагов, +14–22% reasoning-токенов** — и это ДАЖЕ когда файл
   помогает. Механизм вреда: «exploring more, learning less» — файл гонит агента шире шарить по репо.
   → **Каждый блок CLAUDE.md должен отвечать: какое поведение он меняет? Если не меняет — он чистый
   минус 20%.**

3. **🕐 Главный рычаг деградации — глубина сессии, а не размер файла.**
   McMillan: OR = 0.944 на функцию (**−5.6% шансов соблюдения за шаг**), воспроизводится на 2-й
   кодовой базе и на Opus 4.6. Всё структурное — ноль.
   → **Это переворачивает дизайн модуля:** лечение — **ре-инжекция / освежение правил по ходу
   сессии** (у нас уже есть хуковая машинерия — SessionStart, PostToolUse!), а НЕ укорачивание файла.
   Возможно, самый ценный практический вывод всего прогона.

4. **🎯 Дорого — «много правил В ОДНОМ ответе», а не «много правил в файле».**
   ManyIFEval: prompt-level accuracy Claude 3.5 Sonnet **0.95 → 0.48** при 1→10 одновременных
   инструкций. IFScale: **per-instruction** точность держится ~98% до 150.
   → **Считать надо не размер CLAUDE.md, а число ВСЕГДА-АКТИВНЫХ правил, обязанных сработать
   одновременно.** Условно-триггерные — дёшевы. Наши always-on (ITP-маркер, нет-эмодзи,
   нет-двоеточия, «Короткий ответ», Autoflow) — вот дорогая корзина.

5. **🧹 Декоративные строки — измеренный мусор.**
   SPRIG: компоненты класса «good property» («ты — эмпатичный/внимательный ассистент») выбираются
   оптимизатором **реже случайного**, хотя это самый популярный дефолт. Ценность — в CoT/role.
   → Рез мотивационно-декоративного слоя обоснован **напрямую**.

6. **📈 Системный слой как таковой ценен — не минимизируй его в ноль.**
   SPRIG: **~+10%** от оптимизированного системного промпта, переносится между моделями/языками.
   Anthropic: **«minimal does not necessarily mean short»**.

### 4.3 Что это значит для модуля «умного добора информации из сети»

- **Приоритет №1 — детектор НОВИЗНЫ, а не компрессор.** Прежде чем добавить блок в контекст, спроси:
  «достижимо ли это grep-ом/чтением файла за 1–2 шага?» Если да — не добавляй, добавь **указатель**.
  Это прямо следует из ETH-эксперимента с удалением документации.
- **Приоритет №2 — тайминг ре-инжекции.** Раз распад внутрисессионный (OR 0.944/шаг), модуль должен
  **доливать** ключевые правила по глубине сессии, а не отдавать всё на SessionStart.
- **Не строй прокси-метрику качества контекста на perplexity/NLL.** Agarwal: NLL **не предсказывал**
  падение downstream-успеха после 125 шотов. Мерь задачу.
- **Ходи в первоисточник.** Пойманная в §1.3 склейка двух статей поисковиком — ровно тот дефект,
  который модуль обязан не воспроизводить.
- **Флагуй ангажированность источника** (Chroma/RAG-вендоры vs длинный контекст).

### 4.4 Собственные тесты, которые надо прогнать (для HYPOTHESIS-находок)

| # | Тест | Что решает |
|---|---|---|
| **T1** | **Redundancy-ablation на нашем репо.** Для каждого блока CLAUDE.md проверить: восстанавливает ли агент тот же факт через grep/glob/read за ≤2 вызова. Резать **только** восстановимые. A/B на фиксированном сьюте задач (метрика: успех + шаги + токены). | Переносит ETH-эффект (+4% / −2% / +2.7%) на нас. Это ЕДИНСТВЕННЫЙ критерий реза, который подпёрт. |
| **T2** | **Session-depth decay probe.** 3–5 канареечных правил (ITP-маркер первой строкой; нет эмодзи; нет двоеточия перед тулом) — замерить соблюдение как функцию глубины сессии (число тулколов/функций). | Проверяет OR≈0.944 у нас. Если воспроизводится → строим **хук ре-инжекции**, а не режем файл. |
| **T3** | **Simultaneity audit.** Посчитать правила, обязанные сработать в ОДНОМ ответе. Сократить именно **always-on** корзину, условные — оставить. | Переносит ManyIFEval (0.95→0.48 при 10). Даёт правильную единицу измерения «дороговизны». |
| **T4** | **Size-null проверка на НАШИХ объёмах.** McMillan не раскрыл уровни «большого» файла; наш CLAUDE.md (global+project) может быть на порядок больше его «large». Прогнать: полный контекст vs −50% объёма (при СОХРАНЁННОЙ новизне, т.е. режем только дубли) — есть ли разница. | Отделяет «размер не важен» от «размер не важен в диапазоне, который он тестировал». |

## 5. Честные ограничения этого прогона

1. **Не смог прочитать arXiv:2509.14744** (корпус реальных CLAUDE.md, 253→2 303 файла): PDF отдался
   бинарём. **Распределение размеров реальных CLAUDE.md мне неизвестно** — нет референса «где мы на
   фоне популяции».
2. **arXiv:2605.10039 (McMillan) — не извлёк ОПЕРАЦИОНАЛЬНЫЕ УРОВНИ переменных.** HTML — 404, PDF —
   бинарь; абстракт уровней не даёт. **Я не знаю, что там значит «большой файл» в токенах.** Это
   критично: affirmative-null на размер может относиться к диапазону, много меньшему нашего. Поэтому
   пометил HYPOTHESIS, не RULE, и завёл тест T4.
3. **McMillan: одноавторский, не рецензированный, НЕ преregistered препринт**; главный эффект
   (внутрисессионный распад) найден **на этапе анализа**, а не предзаявлен — автор честно это пишет,
   но это снижает вес. Метрика — **одна тривиальная аннотация**, т.е. работа **не проверяет** тезис
   «большой файл ломает сложное многоправильное поведение».
4. **arXiv:2602.11988 (ETH) — препринт.** Рецензирование подтвердить не удалось (извлекатель написал
   «presented at ICML» для февральского препринта 2026 — не верю, **не утверждаю венью**). Эффекты
   **малы (±0.5–4%)**, авторы сами называют дельты «marginal»; **статистическую значимость +4% я не
   подтвердил** по первоисточнику.
5. **Поисковая выдача склеила две разные работы** (§1.3), приписав числа ETH статье Lulla et al. Я
   поймал это только сверкой первоисточников. **Возможно, в корпусе прогона 1 эта склейка уже сидит.**
   Стоит проверить.
6. **Противоречие ETH ↔ Lulla по стоимости не разрешено** (+20% инференса vs −20% output-токенов /
   −28% времени). Мои объяснения (разные метрики: полная стоимость с reasoning vs output-токены;
   бенчмарки vs реальные PR) — **гипотеза, а не установленный факт**.
7. **Пер-модельные ячейки arXiv:2510.05381 по фронтир-моделям извлеклись НЕНАДЁЖНО** (дублирующиеся
   −41.7% на двух разных задачах Claude — артефакт парсинга таблицы). **Не опирался на них**; опираюсь
   только на текстовые утверждения авторов + на атрибуцию крайних значений (85% = Llama-8B/VarSum).
8. **Память агентов (§2.5) — только поисковый уровень**, первоисточники не читал. Не опора.
9. **Не нашёл главного:** **нет ни одной работы, которая сравнивала бы ДЛИННЫЙ, богатый,
   человеком написанный CLAUDE.md (процессные правила, методология — как у нас) против КОРОТКОГО, на
   реальных многошаговых задачах.** Ближайшее — ETH (человеческий vs LLM-генерированный vs никакого,
   но не длинный-vs-короткий) и McMillan (размер, но с тривиальной метрикой и неизвестными уровнями).
   **Этот эксперимент, похоже, просто не существует. Его надо ставить самим (T1/T4).**
10. **Many-shot ICL → наш вопрос: indirectness = large.** Демонстрации ≠ инструкции. Я НЕ утверждаю,
    что «+36.8 пункта на Banking-77» переносится на CLAUDE.md. Я утверждаю более узкое и надёжное:
    **общий закон «больше контекста → хуже» ложен**, а значит, из него нельзя выводить «режь».
    Обоснование реза должно приходить из §1 (прямые работы), а не из «контекст вреден вообще».
