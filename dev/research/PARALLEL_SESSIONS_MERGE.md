## 5. Синтез: ответ на crossover-вопрос для нашего масштаба

Vendor-правило «merge-инфраструктура не для соло» (F4) и открытый вопрос «а если соло гонит 4-5 агентов?» для нас разрешаются данными §3: **contention реален (6.6 merge/день, 3 параллельных PR), но уже канализован** — 90% конфликт-массы падает в два hot-file с работающим протоколом склейки, семантических инцидентов за историю ноль, а сериализованный owner-merge выполняет функцию очереди (человек = merge queue с batch size 1). Поэтому:

- **Полная merge-queue инфраструктура — оверхед, не брать** (подтверждает F4 против F5 для нашего n).
- **Брать нужное подмножество future-state идеи (F5) без инфраструктуры:** verify на предслитом состоянии «ветка + свежий main» непосредственно перед нажатием merge — это единственный незакрытый семантический зазор (см. A1).
- **Самый большой непокрытый класс — ранняя видимость пересечений между живыми полосами** (F2/Clash): сейчас пересечение обнаруживается только в момент merge-конфликта. Read-only `git merge-tree` сторож дёшев и закрывает это до интеграции (B1).
- Потолок параллельности у нас задаётся квотами API (429 на 5-6 полосах), а не merge-риском — WIP-guardrail нужен в коде спавна, не в git (B3).

### Пакет A — текстовые правки экосистемы (все S, полностью обратимы)

- **A1. Autoflow: pre-merge future-state check.** Новый подпункт между «PR готов» и «СТОП перед merge»: если с момента последнего verify в main въехали чужие PR — merge свежего `origin/main` в ветку + `npm run verify` перед нажатием. Закрывает единственный найденный семантический зазор. *(F5; инциденты 0149/0190)*
- **A2. Process triggers: строка «после мерж-волны ≥2 PR — полный verify на актуальном main»** (формализация практики 0190, сейчас ad hoc). *(F1-guardrail)*
- **A3. Autoflow шаг 3: splice-recipe для ROADMAP.md** рядом с DEV_JOURNAL/CHANGELOG: секция «Где мы сейчас» — latest-wins (взять свежую по git log), НЕ склейка обеих сторон. 33 PR/8нед без рецепта — третий по горячести файл. *(эмпирика §3)*
- **A4. Dependency-order note:** при ≥2 открытых связанных PR — явная строка «мержить после #X» в теле зависимого PR (проактивное расширение существующей памяти stacked-PR). *(F5-класс)*
- **A5. Реестр ID-классов в `dev/INFORMATION-MAP.yaml`:** таблица «класс → mint-точка → claim есть/нет → писатели» — прямое противоядие уроку FB-LR-28/29 «guard не самораспространяется». *(грабли §3)*

### Пакет B — механика Кондуктора (репо factory-conductor; отдельный PR там)

- **B1. `factory clash`** — read-only `git merge-tree --write-tree` между парами живых lane-веток, отчёт о пересекающихся файлах/hunks без мутации репо; минимальный первый шаг — filename-overlap подсветка в `factory status`. **S/M.** *(F2/Clash; главный непокрытый класс §5)*
- **B2. Lane-manifest при spawn:** `factory spawn --scope <paths>` пишет декларацию путей в реестр полос; следующий spawn/status варнит о пересечении с живой полосой. **S.** *(F0 hot-files; отбитое «ownership как единственная стратегия» — здесь оно дополнение, не замена)*
- **B3. WIP-лимит константой + quota-preflight:** `MAX_LANES=3` с отказом/очередью при превышении + проверка квоты до спавна судей/полос. Переводит инвариант I-5 из прозы в код. **S.** *(F7; квота-грабли §3)*
