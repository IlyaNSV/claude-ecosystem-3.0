## A. Класс «модуль тестирования пропускает» — главная боль владельца

Четыре прецедента за один день, все пойманы РУЧНЫМ проходом владельца при зелёных гейтах
(P6 GO / P7 STARTS / UJA PASS 7/7 / deploy READY):

- **A1. Задумано дизайном — не построено — невидимо для приёмки** (находка #8, RUN-B.51).
  Dashboard-библиотека: job создан и скачан, dashboard всегда «Пока нет локализаций» —
  list-endpoint не существовал, а UJA j1 принимал empty-state как валидный маркер.
  → **Дельта:** UJA-журнеи авторить из ПОЛНОГО дизайн-состава (все состояния экранов,
  включая with-data), непостроенные состояния — явные SKIP/ESCALATE строки в UJA-отчёте,
  а не молчание. (Частично закрыто в пилоте: j1 теперь list-XOR-empty, j2 ассертит
  появление job — но правило должно жить в КАНОНЕ процесса P8, не в одном пилоте.)
- **A2. Cross-realm матрица не существует** (находка #9, RUN-B.58 — финальная, НЕ чинена
  по решению владельца). Репро владельца: логин в /admin (admin_session) → корневой
  адрес → редирект на /dashboard, который РЕНДЕРИТСЯ как authed-вид (list-fetch получает
  401 и тихо деградирует в empty — выглядит «нормально»), но ЛЮБОЕ действие → 401,
  включая «Выйти» (logout требует user-сессию). Корень-класс: middleware различает
  «есть кука/нет куки», но не РЕАЛМ куки; матрица доступа auth-state×маршрут не имеет
  измерения realm×realm; негативных журнеев класса «admin-кука на user-рутах / user-кука
  на admin-рутах» нет.
  → **Дельта:** канон-директива владельца «матрица has_ui ОБЯЗАТЕЛЬНА» расширяется
  измерением реалмов: состояния = {guest, user, admin, user+admin?} × все руты; из
  матрицы генерятся негативные журнеи; UI-рендер и API-поведение обязаны совпадать
  (тихий degrade в empty при 401 — анти-паттерн: маскирует realm-дыру).
- **A3. Interaction-тупики компонентов** (чип-фикс, RUN-B.50): состояние «выбран и
  недоступен одновременно», валидация загоняет в тупик без пути исправления.
  → **Дельта:** error-states + interaction-переходы — обязательная часть спеки has_ui
  (уже канон-кандидат находки №5 в реестре; чип — второй прецедент класса).
- **A4. Директивы владельца по DoD-входу** (RUN-B.36, подтверждены практикой):
  realistic-input «видео 30 мин – 2 ч» на DoD первого релиза (5-сек — только dev);
  импликации: Whisper ~25MB → chunking, стоимость, время пайплайна.

- **C1. Live-devtools мост** (находка #7, RUN-B.49): console/network браузера владельца

- **C2. Одна хост-сессия на прогон** (RUN-B.44/45, решено владельцем): перехват роли
  «сосед упал» — только после верификации ХОСТ-слоя (свежие коммиты ledger-ветки);
  pull ≠ прочитал. Кандидат в CONDUCTOR.md инварианты.
- **C3. API-нестабильность — штатный режим, не исключение** (7+ инцидентов за день:
  s31 stall 31 мин с молчаливым авторетраем, хост ×3, s34 ×2 «Connection closed
  mid-response», s35 retry-баннеры): recovery-протокол «верифицируй своё состояние →
  продолжай со шва», тактика «пиши файлы частями» на длинных генерациях, поллер-детект
  v5 (активный спиннер ≠ past-tense след; ERROR-STALL = API Error без активного
  спиннера; авторетрай-баннер и фоновые шеллы = busy). Всё валидирует канон
  «hook-relay/events.jsonl, не pane-peek» для H2 — pane-детект потребовал 5 итераций
  критериев за один вечер.
- **C4. Манёвр Esc → RESUME-заметка → ребут → tmux+`--resume`** (RUN-B.54/55):
  живо валидирован — executor-сессии переживают ребут VM без потери нити. Кандидат в
  vm-factory-ops §4 как штатная процедура остановки/продолжения.
- **C5. «Сделай шов» = wind-down сигнал** (урок RUN-B.54): после запроса шва новых
  единиц не начинать без явного «продолжай».
