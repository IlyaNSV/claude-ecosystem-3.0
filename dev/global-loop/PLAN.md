## 0. Цель одним контуром

Единый **global loop** end-to-end: владелец даёт интенцию → **Кондуктор** (пульт)
диспетчеризует релизный цикл нужного проекта на VM → экосистема итерирует по канонам
(P3→P6 → deploy-to-stage) → **DoD-предикат** меряет «релиз выкачен и готов к
использованию» → незакрытые пункты становятся очередью следующей итерации → пульт
надзирает, отвечает в мандате, батчем эскалирует владельцу → зелёный DoD + owner ratify
= выход цикла (`RL.status: released`). Два трека — две половины одного контура:
**DoD — критерий остановки внутреннего цикла проекта; Кондуктор — внешний цикл над
десятком таких проектов.**

**Дефиниция Кондуктора (владелец, 2026-07-15, дословно):** «кондуктор — это
мультипликатор сессий и слой более автономного управления ими». Две оси ценности:
сколько параллельных сессий/проектов позволяет вести × насколько автономнее (в мандате)
они управляются.

**Стратегия входа (владелец, 2026-07-15, дословно):** «Сначала я попробую из этого
проекта (терминала) управлять тестовым проектом с ассистансом, накопим опыт и пойдем в
кондуктора уже после.» ⇒ перед H-фазами — **Волна 0 «Ассист-пульт»** (ручной режим,
опыт → данные для мандата и H0); вход в стройку Кондуктора — по «го» владельца.

## 1-bis. Знаниевая плоскость Кондуктора (требование владельца 2026-07-15)

> «Кондуктор должен знать как работает экосистема еще лучше, чем я, надо подумать как ему
> дать все необходимые для корректной работы описания и документацию + сделать возможность
> хождения и добора информации об экосистеме.»

**Дизайн — SSOT в `factory-conductor/CONDUCTOR.md` §Knowledge plane (K0).** Ядро: знание —
не копия доков (копия = дрейф), а **лестница добора** по уже построенной инфраструктуре
экосистемы: резолвер `dev/INFORMATION-MAP.yaml` («класс вопроса → SSOT → кому верить») →
генерируемые каталоги (команды/глоссарий/скиллы — не дрейфуют by design) → полные SSOT →
живое состояние проекта. Два жёстких правила: **(а) version-skew** — про поведение проекта
добор из его `.claude/` (инсталляция несёт доки своей версии), про канон — из host-клона
ecosystem-репо; **(б) ответ пульта без указателя на SSOT-источник = не ответ** (это и есть
операционализация «знает лучше владельца»: не «помнит больше», а не отвечает по памяти
вообще). Вплетение в волны: H0-бриф включает контрольные вопросы знания (валидация
лестницы на живом sweep), H1 — version-skew механика при диспетчеризации.

**Прогрев хост-сессии (Волна 0; DEC-DEV-0222):** выбранная сессия-кондуктор на старте
длительной работы грузит онбординг-пак — `node dev/meta-improvement/scripts/host-onboard.cjs`
(`--list` — состав/размеры; `--skip`/`--only`/`--budget` — урезание). Пак = префетч ступеней
L0-L2 лестницы (шов+план, CONDUCTOR.md, хартии, INFORMATION-MAP, генерируемые каталоги,
гейты+fabric, VM-грабли) + указатели L3 «для дообучения по требованию»; собирается из ЖИВЫХ
файлов при каждом запуске, на диск не пишется (не зеркало — дрейфовать нечему); бюджет
≤40k ток. (~20% окна — потолок владельца). Оправдан профилем сессий-кондукторов: очень
длинные, контекста генерируют мало — разовый прогрев живёт всю сессию.

## 2. Сшитая фазировка — волны с зависимостями

```
Волна 0 (ассист-пульт, СЕЙЧАС):  R0 + ручное управление тестовым проектом (владелец из
                                 терминала + ассистент) → опыт-ledger ASSIST_LOG.md
        └── gate: «го» владельца по накопленному опыту ──▶
Волна 1 (наблюдатель):             H0 (+K-валидация)
Волна 2 (автоматизация):           R1 ──▶ H1                   ← сшивка №1: sweep читает DoD-отчёт
Волна 3 (событийный контур):  events.jsonl ─▶ H2 ∥ R2         ← сшивка №2: DoD-парковки = вопросы пульта
Волна 4 (масштаб):                 H3 ─▶ H4
Вне волн (event-gated):            R3 (реальный prod-хост)
```

| Волна | Единицы | Пререквизиты | Определения |
|---|---|---|---|
| **0 — Ассист-пульт** (ручной режим) | **R0**: термин DoD в канон + DoD-чеклист в body RL (soft) + ручная проверка — рабочий инструмент ассист-управления. **Ассист-режим**: владелец из терминала ecosystem-сессии управляет тестовым проектом на VM, ассистент готовит команды/разбирает вердикты/следит; **опыт-ledger** `dev/global-loop/ASSIST_LOG.md` (append-only: классы возникших вопросов, эскалации, расход внимания, узкие места) — это данные для мандата v0, H0-брифа и приоритетов H-фаз | нет | R0-R3 — `dev/release-dod/TRACK.md` §6; H0-H4 — `factory-conductor/CONDUCTOR.md` §roadmap |
| **1 — Наблюдатель** | **H0**: спайк-наблюдатель — пре-регистрация брифа (что меряем, $/sweep, stop-критерии, K-валидация; образец — `meta-system/SPIKE_M1_BRIEF.md`) → read-only sweep → отчёт | **«го» владельца по опыту Волны 0** | |
| **2 — Автоматизация** | **R1**: `/product:release-dod` (read-first агрегатор + отчёт ✅/🟠/❌) → **H1**: диспетчер (intent → spawn/send на 1 проекте; вендоринг thin-waist схем меты). Сшивка №1: sweep H1+ включает DoD-отчёт | Волна 1 | |
| **3 — Событийный контур** | **events.jsonl** (хуки полосы → durable timeline; детализация — README factory-conductor §роадмап) → **H2** (Q&A в мандате) ∥ **R2** (loop-wiring: незакрытые DoD-пункты → PA/owner-queue; fabric-ячейка `release-dod`). Сшивка №2: DoD-парковки — класс вопросов, которые пульт batch-эскалирует | Волна 2; H2 жёстко gated на events.jsonl (Q&A по pane-peek запрещён) | |
| **4 — Масштаб** | **H3** (флот: реестр, WIP/внимание-бюджет, единая очередь) → **H4** (deviation watch + strengths/weaknesses ledger → feedback-outbox) | H1+H2 на 2 проектах | |

**Гейт-дисциплина:** перед каждой волной — `readiness-gate.md`; live-прогоны на VM — по
`checklists/live-run-validation.md` (executor/reviewer separation); после постройки, до
веры в интеграцию — `smoke-test-plan.md`. Смоук-долг >2 непрогнанных планов = стоп новым
built (инвариант меты №1).
