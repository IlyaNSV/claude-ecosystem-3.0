# Contract Design — Skill for Integrator

Design a structured contract between a **producer** (artifact-emitting tool/module) and a **consumer** (artifact-consuming tool) — and instantiate the adapter script that performs the transformation.

> **Manual in v1 (per ROADMAP Phase 5 risk note + DEC-DEV-0040).** Generation algorithm formalizes later (Phase 7+) once 2-3 adapters are written by hand and patterns emerge. Don't try to over-generalize from a single example.

## When invoked

- During `/integrator:add <tool>` **Stage 5 (contract)** — for each (producer, consumer) pair created by adding this tool
- During `/integrator:update <tool>` **Stage 4 (contract repair)** — when version bump introduces schema drift
- During `/integrator:debug` — when contract is suspected as root cause of cross-tool failure

### Inline vs subagent invocation

| Caller | Mode | Why |
|---|---|---|
| Main `/integrator:add` orchestration | Subagent (`contract-designer`) | Producer artifact sample + consumer capabilities + transformation logic = heavy context that bloats main session; isolated context preserves UX flow |
| Main `/integrator:update` contract repair | Inline | Most repairs are field-level diffs; full contract redesign rare |
| Main `/integrator:debug` | Inline | Targeted check against one contract's failure modes |

## Contract artefacts

A contract has **two files** (paired):

| File | Purpose | Generated by |
|---|---|---|
| `.claude/integrator/contracts/CNT-NNN.yaml` | Machine-readable schema per SPEC §5.1 | Subagent / inline |
| `.claude/integrator/contracts/CNT-NNN.md` | Human-readable companion: rationale, examples, edge cases, troubleshooting | Subagent / inline |

Adapter script lives separately: `.claude/integrator/adapters/<adapter-name>.js` (instance), copied from `adapters/<adapter-name>.js` (repo reference) for known producer/consumer pairs — see "Adapter instantiation" below.

## Process

### Step 1 — Identify producer + consumer

From the active-tools.yaml + tool profile, determine:
- **Producer** — the module/tool emitting the artifact (e.g., `product-module`)
- **Consumer** — the tool that needs the artifact transformed (e.g., `cc-sdd`)
- **Artifact in flight** — what's being passed (e.g., `handoff.md`)

If both ends are external tools (not Product Module), the producer's tool-profile must declare an output artifact that matches the consumer's input artifact (by type or schema). If not — that's a coverage gap, not a contract; surface to main session.

### Step 2 — Read producer artifact sample + consumer input requirements

- Producer side: fetch a real sample if available (e.g., existing `.product/handoffs/FM-001-handoff.md`); otherwise use a fixture (`tests/fixtures/`).
- Consumer side: from tool profile `inputs[]` — what format does the consumer expect? schema ref? adapter required?

Document field-by-field correspondence: which producer field maps to which consumer field. Note **gaps**: producer-only fields (drop), consumer-only fields (must derive or default).

### Step 3 — Choose transformation type

Per SPEC §5.1 `transformation.type`:

| Type | When | Implication |
|---|---|---|
| `direct` | Producer artifact is already in consumer's input format | No adapter needed; contract documents the pass-through |
| `adapter_script` | Field/shape transformation needed; reproducible | Adapter file (`adapters/<name>.js`); preferred for D2-Tech pipelines |
| `manual` | One-off / requires human judgement | Documented procedure in contract.md; flag for v1.1+ formalization |

**Default for D2-Tech pipelines:** `adapter_script`.

### Step 4 — Check for existing reference adapter

Before writing custom code: check `adapters/` (repo top-level) for a reference adapter that matches the (producer, consumer) pair.

```bash
ls adapters/<producer>-to-<consumer>.js 2>/dev/null
```

If exists → **use the reference** (per DEC-DEV-0040 Q1 dual-location pattern). Skip to Step 7 (adapter instantiation).

If not → proceed to Step 5 (custom adapter design).

### Step 5 — Design custom adapter (if no reference exists)

Follow `adapters/handoff-to-ccsdd.js` as the canonical structural template:

1. **Header metadata block** (JSDoc comments): `@target_tool`, `@target_tool_version`, `@contract_schema_version`, `@source_ref`, `@installed_at`. Repo reference leaves last 3 blank; install populates them.
2. **`--verify-only --fixture <path>`** mode mandatory — for Stage 6 contract-test without invoking the actual consumer.
3. **Node stdlib only** — no npm deps in reference adapters; cross-platform LF-normalized I/O.
4. **Line-based frontmatter parsing** — NOT regex on multi-entry blocks (lesson DEC-DEV-0031 A1).
5. **Contract validation function** — returns `{ passed: boolean, checks: [{id, level, status, detail}] }` with `C-NN` check IDs per contract.
6. **Transformation function** — produces the consumer's input shape; embeds `contract_schema_version`.
7. **Exit codes:** 0 contract OK, 1 contract validation failed, 2 parse error.

Place new reference adapter at `adapters/<producer>-to-<consumer>.js` (it will be copied to `.claude/integrator/adapters/` on install).

### Step 6 — Write the CNT-NNN.yaml + CNT-NNN.md pair

Use the canonical CNT YAML template (Step 8 below). Companion .md narrates:

- **Why this contract exists** (which pmo-mapping zone necessitates it)
- **Field-by-field mapping table** (producer → consumer)
- **Edge cases tested** (e.g., partial handoff, missing optional field)
- **Failure mode runbook** (what's the symptom, what's the fix, which journal entries to consult)

### Step 7 — Adapter instantiation (copy reference → instance, inject metadata)

For both reference-reused and newly-written adapters, the install step is identical:

1. `cp adapters/<adapter>.js .claude/integrator/adapters/<adapter>.js`
2. Inject metadata header values:
   - `@target_tool_version: <from tool profile>`
   - `@source_ref: <git rev-parse HEAD>` (commit hash of reference at install time)
   - `@installed_at: <ISO-8601 now>`
3. Run smoke verify: `node .claude/integrator/adapters/<adapter>.js --verify-only --fixture tests/fixtures/<fixture>.md` — must exit 0.

`/integrator:update` later diffs installed metadata against repo HEAD + new tool version to detect drift.

### Step 8 — Canonical CNT-NNN.yaml template (B.1 frontmatter convention)

> **Use exactly these field names.** AI-prone variants forbidden: see Anti-patterns below.

```yaml
contract:
  id: CNT-NNN                              # zero-padded; assign next free
  name: "<Producer artifact> → <Consumer command>"
  producer: <module-or-tool-id>            # e.g., product-module, cc-sdd
  consumer: <tool-id>                      # e.g., cc-sdd, beads
  created: YYYY-MM-DD
  last_verified: YYYY-MM-DD                # last successful adapter run
  status: active | broken | draft | deprecated

data_flow:
  from:
    artifact: <artifact-type>              # e.g., product-handoff.md
    location: <path glob>                  # e.g., .product/handoffs/FM-{NNN}-handoff.md
    format: <markdown | yaml | json | other>
  to:
    artifact: <consumer input type>        # e.g., spec-init input
    location: <consumer-internal | path>
    format: <format>

transformation:
  type: direct | adapter_script | manual
  script: <relative path>                  # for adapter_script; e.g., .claude/integrator/adapters/handoff-to-ccsdd.js
  contract_schema_version: 1               # mirrors adapter's CONTRACT_SCHEMA_VERSION
  adapter_description: |
    <2-4 lines: what adapter does at a high level>

validation:
  pre:
    - check: "<precondition assertion>"
  post:
    - check: "<postcondition assertion>"

failure_modes:
  - symptom: "<observable failure>"
    likely_cause: "<diagnosis hint>"
    action: "<remediation command>"
```

### Step 9 — Update pmo-mapping.yaml + tool active-tools.yaml

After contract is created/updated:

- Add CNT-NNN reference to relevant `pmo-mapping.yaml` zone's `contracts: []` list
- Add CNT-NNN to producer/consumer entries in `active-tools.yaml` (so removing a tool surfaces dependent contracts in impact analysis)

### Step 10 — Journal entry (autologged by journal-hook)

Contract creation triggers `journal-hook` (Phase 5.F) — entry tagged `#contract-design` with CNT-NNN, producer, consumer, transformation.type. Adapter manually adds rationale + lessons section.

## Anti-patterns

1. **Field name drift (B.1).** Use canonical contract schema verbatim. Forbidden variants:
   - `contract.id` — NOT `contract_id`, `cnt_id`
   - `data_flow.from / to` — NOT `source / target`, `input / output`
   - `transformation.type` — NOT `adapter_type`, `kind`
   - `transformation.script` — NOT `adapter`, `adapter_path`
   - `failure_modes[].symptom / likely_cause / action` — NOT `error / cause / fix`
2. **Skipping reference-adapter check (Step 4).** Custom adapter when reference exists = duplicate maintenance burden + drift risk.
3. **Inventing CNT IDs out of sequence.** Assign next free integer; check existing `.claude/integrator/contracts/` first.
4. **Embedding consumer-specific business logic in adapter.** Adapter does shape transformation only; business decisions stay in Product Module.
5. **Forgetting `--verify-only` mode in custom adapter.** Stage 6 verify cannot run without it; contract is unfinished.
6. **Optimistic `confidence: high` on transformation.** Until fixture verify + at least one real run, transformation confidence is `medium`.
7. **Composing contracts in main session context.** Use `contract-designer` subagent for fresh isolated context — main session focuses on UX orchestration.
