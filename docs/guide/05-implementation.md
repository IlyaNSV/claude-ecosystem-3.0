# От handoff к коду — Integrator + Orchestrator

> **Для кого:** у тебя есть готовый `handoff.md` одной фичи (см. [`01-first-session.md`](01-first-session.md) §7), и ты хочешь превратить его в реализованный и проверенный код. Этот документ — про вторую, делегируемую половину пайплайна (D2-Tech + D3): подключить реализатор и прогнать оркестратор.

---

## 0. Предусловие — подключить реализатор

Оркестратор не пишет код сам — он дирижирует внешним инструментом-реализатором (по умолчанию **cc-sdd**, spec-driven).

**Что такое cc-sdd:** сторонний open-source тулкит spec-driven development для Claude Code. Он ставит в `.claude/` проекта набор kiro-команд и скиллов (`/kiro:spec-init`, `kiro-spec-batch`, steering и др.), которые превращают требования в спеки → дизайн → задачи → код. Отдельная регистрация, аккаунт или API-ключ **не нужны**: команда ниже сама скачает, установит и подключит его под управление Integrator (профиль инструмента, контракты CNT-*, адаптер handoff→spec), показав approve-гейт перед установкой. Один раз подключи его:

```
> /integrator:add cc-sdd
```

> ⚠️ Без **активного** cc-sdd `/orchestrator:run` остановится на pre-flight. Это первое, что он проверяет.

---

## 1. Четыре процесса

`/orchestrator:run <process>` запускает один процесс как in-harness Workflow. Порядок: **P3 → P4 → P5 → P6**.

> **Почему P3–P6, а не P1–P7.** Полное семейство рантайм-процессов Orchestrator — `P1–P7` (P1 init · P2 decide-architecture · P3 batch→cc-sdd · P4 audit-fidelity · P5 tdd-impl · P6 validate-impl · P7 runtime-smoke). Оператор запускает середину — **`P3–P6`**: P1/P2 — внутренняя подготовка Orchestrator, а **P7 runtime-smoke прогоняется как детерминированная нога внутри вердикта P6**, не отдельной командой. Не путать с процессами Product `P1–P5` и семью *принципами* `P1–P7` — полная разводка осей в [`03-glossary.md`](03-glossary.md) «Оси именования».

| Процесс | Что делает (по-человечески) | Что возвращает |
|---|---|---|
| **P3** `batch-features-to-cc-sdd` | Превращает handoffs в спеки cc-sdd (kiro-specs) | `coverage_incomplete`, `blocked` |
| **P4** `audit-spec-fidelity` | До имплементации сверяет спеку с `.product` (нет ли дрейфа, фабрикаций, непокрытых модулей) | `impl_ready`, `spec_fixed`, `product_routed`, `residual`, `coverage_gaps` |
| **P5** `feature-to-tdd-impl` | Имплементирует фичу через TDD, по-задачно коммитит; сам делегирует финальный гейт P6 | `implemented`, `blocked`, `conflicts`, `concerns`, `readiness` |
| **P6** `validate-feature-impl` | Фичевый GO/NO-GO **после** impl: полный прогон тестов + сборка + 3 валидатора (покрытие требований / соответствие дизайну / межзадачные стыки) + verify-finding | `result`, `readiness`, `confirmed_findings`, `conflicts` |

P6 запускается из P5 автоматически, но его можно прогнать и отдельно — пере-гейтнуть уже реализованную фичу.

---

## 2. Последовательность

```
> /orchestrator:run batch-features-to-cc-sdd --all              # P3
> /orchestrator:run audit-spec-fidelity --feature FM-003        # P4  (между P3 и P5)
> /orchestrator:run feature-to-tdd-impl --feature FM-003        # P5  (→ P6 авто)
```

`--all` — по всем готовым handoffs; `--feature FM-NNN` — по одной фиче.

---

## 3. Как читать вердикт

Исход гейта — это **две независимые оси** плюс список конфликтов. Не путай «код плох» и «гейт не смог судить»:

- **`result`** — *хорош ли код*: `GO` · `NO-GO` · `MANUAL_VERIFY_REQUIRED`
- **`readiness`** — *смог ли гейт вообще судить*: `READY` · `DEGRADED` · `ENV_NOT_READY`
- **`conflicts[]`** — cross-spec/design противоречия; **непустой список понижает `GO` до `MANUAL_VERIFY`**

| `result` | `readiness` | Что это значит | Что делаешь ты |
|---|---|---|---|
| **GO** | READY | Код готов, гейт судил полноценно | Принять (но сверь §4 — на чём гонялись тесты) |
| **GO** | DEGRADED | Готов; часть среды была частично недоступна | Принять с оговоркой, отметить деградацию |
| **MANUAL_VERIFY** | ENV_NOT_READY | **Субстрат был недоступен** (БД/Docker/Redis off) — это НЕ значит, что код плох | Поднять среду, перепрогнать P6 |
| **MANUAL_VERIFY** | READY/DEGRADED | Есть остаточные находки или `conflicts` | Разобрать `confirmed_findings` / `conflicts` |
| **NO-GO** | любая | High-severity дефект остался после 3 раундов починки | Фича не готова — смотреть findings |

**Про `conflicts[]`:** если починка упёрлась в противоречие *между* спеками/требованиями или в само-противоречие дизайна — оно **эскалируется тебе** (или автору спеки), а не «решается» агентом в одну сторону. Решение тут продуктовое/архитектурное. Запись с `masked:true` означает, что починка сообщила об одностороннем разрешении — проверь, что конфликт не замяли.

---

## 4. GO ≠ всегда «всё по-настоящему»

**Обязательное раскрытие субстрата.** Если acceptance/E2E гонялись на **Mock/stub** вместо отложенного реального стыка (провайдер, внешний API, секрет, адаптер) — сводка GO обязана это сказать: «E2E на Mock-провайдерах; реальные адаптеры — неподключённые скелеты; реальный доступ отложен».

Зелёный «feature complete / GO» над неподключённым реальным стыком — это over-claim. Смотри список **`concerns[]`** и его маршрут:
- доступ / инструмент / секрет → **Integrator**;
- выбор провайдера → **Product**.

GO с такими concerns — это **GO-с-оговорками**, не «всё готово по-настоящему».

---

## 5. Где выходы

- Покоммитная имплементация (sha задач — в сводке прогона).
- Эскалации и отложенное (concerns, conflicts, product-routed дрейф) → `.claude/pending-actions.md`.
- **Один orchestrator-workflow на репозиторий за раз** (FB-004) — не запускай два процесса, пишущих в одно дерево, параллельно.

## 6. Линия между сессиями — Process Fabric (opt-in)

Если фича идёт через несколько сессий, добавь `--fabric` к первому прогону — линия «P3 → P4 → P5/P6 → P7 → деплой (E.B) → приёмка user-journey (P8)» получит машинную память: после каждого прогона движок вычисляет, куда линия перешла, паркует её на человеческих гейтах (через `pending-actions.md`) и предписывает следующий шаг; свежая сессия начинает с машинного «где мы». Как читать `status`/owner-queue и разрешать гейты — [`07-fabric.md`](07-fabric.md).

## 6.5. P8 · приёмка user-journey (после деплоя, перед `done`)

**Когда.** Fabric-ячейка **после** успешного staging-деплоя (`DEPLOYED × READY`, healthcheck 2xx) и **перед** `done`. Диспетчеризуется существующей `/orchestrator:run user-journey-acceptance` (обычно предписывает fabric, руками запускать не нужно).

**Что даёт.** P7 отвечает «стартует ли процесс?» (HTTP-liveness), но не «работает ли **первое касание пользователя**?». Живой прецедент: P6 GO, P7 READY, деплой DEPLOYED и `/health` 2xx — а пост-логин у пилота падал в 404. P8 закрывает разрыв: гоняет **детерминированные Playwright-журнеи** (`tests/uja/*.spec.ts`, один spec-файл = один журней) headless против живого staging-URL с реалистичной имитацией пользователя. Вердикт — **байт-редукция по JSON-репорту** (`uja-report.cjs`), не суждение LLM.

**Как читать вердикт `uja_result`:**

| `uja_result` | Что значит | Куда идёт линия |
|---|---|---|
| **PASS** | Все журнеи зелёные — пользователь проходит поток | → `done` (фича доставлена) |
| **FAIL** | Журней сломался (первое касание битое) | → парковка `awaiting_journey_fix`; владелец ре-драйвит починку через `implementing` (гейт **не** чинит сам — capture-don't-fix). `journeys_failed[]` называет сломанные |
| **ENV_NOT_READY** | DoR-разрыв: Playwright не подключён / нет журнеев / staging не 2xx / **0 журнеев в репорте** | → `runtime_gate_retry`; закрой разрыв (`/integrator:add playwright`, напиши журнеи из NM, подними staging) и перепрогони |

**Правило нуля-доказательств:** репорт с 0 журнеев — это `ENV_NOT_READY` (не смог судить), **никогда не PASS** — зелёный гейт над «ничего не прогнали» = ложный зелёный. Скриншоты/трейс шагов лежат в `artifacts_dir` — это visual-conformance доказательство для ревью владельцем против дизайн-MK (авто-diff MK — v1.1). **Бюджет:** журнеи бьют по **реальному** задеплоенному приложению — держи их на минимальных фикстурах (расход реального вендора). prod-журнеи — вне scope v0.

## 7. Если блокирует

| Сигнал | Что значит | Действие |
|---|---|---|
| `conflicts[]` непусто | Противоречие спек/дизайна | Решение владельца; **не патчить вокруг** |
| `readiness=ENV_NOT_READY` | Субстрат лежал | Поднять БД/Docker/Redis → перепрогнать |
| `NO-GO` / `MANUAL_VERIFY` | Фича не готова | Разобрать `confirmed_findings` |
| P3 `coverage_incomplete` | Спека не покрыла часть id | Перепрогнать нужный `kiro-spec-*` |
| P4 `residual` / `coverage_gaps` | Дрейф спеки / модуль, который никто не строит | Поправить спеку / добавить задачу → перепрогнать P4 |
