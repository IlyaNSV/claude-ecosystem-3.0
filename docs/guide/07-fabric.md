# Process Fabric — линия фичи, которая переживает сессии

> **Для кого:** ты гоняешь фичи через `/orchestrator:run` (см. [`05-implementation.md`](05-implementation.md)) и хочешь, чтобы прогресс «P3 → P4 → P5/P6 → P7» не жил у тебя в голове между сессиями: где линия остановилась, чего она ждёт, что запускать следующим. Fabric — машинная память и штурман этой линии.
>
> **Одной фразой:** после каждого прогона движок принимает его результат, детерминированно вычисляет, куда линия перешла, и **предписывает** следующий шаг — а новая сессия начинает с машинного «где мы», вместо реконструкции по хвостам документов.

---

## 1. За 60 секунд

- **Fabric — это** тонкий слой координации *между* процессами: charter (декларативная карта состояний линии) + движок `fabric-engine.cjs` (event-sourced) + актуаторы (диспетчер `/orchestrator:run`, PA-журнал, SessionStart-инжект).
- **Fabric — это НЕ**: он не вызывает LLM, не запускает процессы сам, не заменяет Workflow внутри процесса и не авто-аппрувит гейты. Он только помнит состояние и предписывает — исполняет предписание диспетчер (и видишь это ты).
- **Opt-in.** Линия заводится флагом `--fabric`; без него всё работает как раньше. Выключенный Fabric = сегодняшнее поведение, ничего не ломается.

---

## 2. Ключевые понятия

| Понятие | Что это | Где живёт |
|---|---|---|
| **Charter** | Декларативный statechart линии (JSON, подсет XState-семантики): состояния, события, какой процесс запускается в состоянии (`invoke`) и как его результат превращается в события (`ingest`-маппинг) | `.claude/orchestrator/charters/*.json` |
| **Инстанс** | Одна конкретная линия («FM-006 в производстве»): текущее состояние + полный журнал событий | `.claude/orchestrator/fabric/<instance-id>/` |
| **Событие** | Материализация structured-результата прогона (поля result-файла → `evt:…`). Никогда «LLM решил, что событие случилось» | `events.ndjson` (первичный источник истины; `state.json` — снапшот) |
| **Prescription** | Предписание следующего шага: `run-process` / `human-gate` / `wait` / `none` — с F1-диспозицией (auto / human-gate) | stdout `tick`/`ingest` |
| **Owner-queue** | Единая приоритизированная очередь твоих гейтов со всех линий (floor > conflict > gate > review) | `owner-queue.json`, показывает `status` |
| **Лимиты (WIP)** | Backpressure: одна orchestrator-линия на репозиторий (FB-004) — отклонённый старт это очередь, не ошибка | `limits.json` |

Первый (и пока единственный) charter — `feature-production-line`: сквозная линия фичи `handoff → authoring (P3) → fidelity-гейт (P4) → имплементация (P5/P6) → runtime-гейт (P7) → done`, с парковками на человеческих гейтах.

---

## 3. Как завести линию

```
> /orchestrator:run batch-features-to-cc-sdd --feature FM-006 --fabric
```

Диспетчер сам сделает `init` + `evt:line.start` и дальше **после каждого прогона** скормит результат движку (`ingest`). Тебе остаётся читать предписания:

- `run-process` + `disposition: auto` → диспетчер запускает следующий процесс сам (полным брекетом);
- `human-gate` → линия паркуется и ждёт тебя (§5);
- `none` + `final` → линия завершена.

Если старт отклонён (`wip_ok` guard) — в orchestrator-lane уже есть живая линия: доведи или закрой её (`status` покажет), вторую не форсируй.

---

## 4. Как читать status и owner-queue

```
node .claude/orchestrator/lib/fabric-engine.cjs status        # живые инстансы
node .claude/orchestrator/lib/fabric-engine.cjs status --all  # включая завершённые
```

Что смотреть:

- **инстансы** — id, subject (фича), текущее состояние, финальность;
- **owner-queue** — единая очередь твоих решений по приоритету. Если тут что-то есть — линии ждут *тебя*, а не наоборот;
- состояния-парковки: `awaiting_product`, `awaiting_capability`, `escalated`, `runtime_gate_retry` — линия стоит на человеческом гейте.

Кроме того, при старте любой сессии в проекте сводка живых инстансов + топ owner-queue **инжектится в контекст автоматически** (SessionStart-хук `session-fabric-status.js`; warn-only, выключается `FABRIC_STATUS_INJECT=0`). Нет инстансов — хук молчит.

---

## 5. Человеческий гейт: разрешить и продолжить (PA-мост)

Когда линия паркуется на гейте, движок уже спроецировал его в канонический `.claude/pending-actions.md` как запись `PA-NNN` с машинными маркерами (`fabric-instance` / `fabric-state` / `resume-event`). Дальше:

1. **Реши вопрос по сути** (продуктовая развилка, конфликт спек, capability-запрос — сам гейт объясняет, что решается).
2. **Флипни `Status` этой PA на `done`** в `pending-actions.md`.
3. В любой следующей сессии:

```
node .claude/orchestrator/lib/fabric-engine.cjs pa-scan --at "<ISO-now>" --tick
```

`pa-scan` найдёт разрешённые fabric-PA, сматчит их с запаркованными инстансами и тикнет записанный `resume-event` — линия поедет дальше (обычно это делает диспетчер, просить руками не обязательно). PA со статусом `dismissed` только *показывается* — abort или resume решаешь ты явно.

## 6. Дисциплина и страховки (что машина не даст сделать)

- **Только полный брекет.** Событие-результат процесса принимается через `ingest --run-id` (тот же result-файл, что у `run-ledger finish`). Голый `tick` такого события движок **откажет** (bracket-guard): нельзя «прошагать» автомат руками мимо реального прогона. Осознанный обход — `--force-manual "<причина>"`, причина навсегда штампуется в аудит-журнал события.
- **Идемпотентность.** Повторный `ingest` с тем же `run-id` — no-op: упавший и перезапущенный брекет не тикнет линию дважды.
- **Floor непробиваем.** Каждое предписание проходит autonomy-policy (F1); `prod_deploy` / `destructive` / `spend_money` / `provision_real_secret` не переводятся в auto никаким флагом.
- **Восстановление после обрыва:**

```
node .claude/orchestrator/lib/fabric-engine.cjs replay --instance <id>
```

пересобирает состояние из `events.ndjson` и сверяет со `state.json` (exit 2 = снапшот дрейфанул). `state.json` руками не править — источник истины всегда журнал событий.

## 7. Как добавить процесс в линию (или новый charter)

Charter — исполняемый артефакт, а не описательный: дрейфовать от реальности он может только там, где дублирует её. Поэтому правило одно — **ingest-маппинг привязывается к фактическим полям result-файла процесса** (тем самым, что процесс возвращает в `run-ledger finish`).

Чеклист:

1. **Состояние** в `states`: `invoke.process` = имя существующего Workflow-процесса; `meta.determinism` (DL0–DL3/H) и `meta.autonomy` (`auto` / `human-gate`).
2. **Переходы** в `on`: по одному `evt:…` на каждый значимый исход; массив кандидатов = «первый прошедший guard выигрывает» (так выражаются bounded-циклы: счётчик + guard `rounds_left`).
3. **`ingest`-маппинг**: правила `when {path, eq|in|nonEmpty} → emit` по реальным полям результата. Проверь их против фактического возврата процесса (`.mjs` return), не против желаемого.
4. **Человеческие исходы** — в состояние-парковку с `actions: ["queue_owner"]` (гейт попадёт в owner-queue и в PA-журнал автоматически).
5. **Юниты** в `tests/orchestrator/fabric-engine.test.cjs`: переход, ingest-маппинг, replay == snapshot. `npm run verify` зелёный.

Референс — действующий charter [`orchestrator/charters/feature-production-line.json`](../../orchestrator/charters/feature-production-line.json); дизайн-решения и формат целиком — `dev/process-fabric/CONCEPT.md` §4. Новые charter'ы заводятся по live-нужде, не впрок (их мало — это осознанно).

---

## 8. Честные границы

- **Live-валидировано** (2026-07-10, VM-пилот): линия прожила ≥2 сессии с restore; NO-GO → remediation → GO прошёл машинно; человеческий гейт разрешён через owner-queue и продолжил линию.
- **Не валидировано live:** ветка `runtime_gate_retry` → `evt:env.up` (оба захода замаскировались upstream-условиями); известный долг — P7-probe даёт false-negative на pnpm-monorepo.
- **Между сессиями Fabric «спит».** Он не демон: события доставляют сессии, хуки и ты. Durable здесь состояние, а не event-loop.
