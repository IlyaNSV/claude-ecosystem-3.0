# PS — Problem Statement

> **Тип:** problem-statement
> **Домен:** D1 — Product Discovery & Strategy
> **Review:** 🟠 Strategic
> **Cardinality:** Singleton (1 на продукт)
> **Владелец:** Product Module / процесс P1 Discovery

## Purpose

Корневой артефакт продукта. Формулирует **проблему**, которую продукт решает, **контекст** её возникновения и **последствия** для целевой аудитории. Является источником для всех остальных артефактов — изменение PS означает потенциальный pivot.

## Frontmatter Schema

```yaml
---
id: PS                              # Фиксированный, без номера
type: problem-statement
title: "Короткая формулировка проблемы"
status: draft | active | pivoted | archived
confidence: high | medium | low                  # C2 modification — обязательно
confidence_notes: "string"                       # required если confidence != high
created: YYYY-MM-DD
updated: YYYY-MM-DD
version: 1                           # инкрементируется при каждом active-переходе
---
```

## Body Structure

Обязательные секции:

1. **Problem.** Что не работает / больно / неэффективно для целевой аудитории.
2. **Context.** В каких ситуациях проблема возникает. Частота, триггеры.
3. **Current alternatives.** Как люди решают эту проблему сейчас (включая «никак»).
4. **Consequences.** Что случается, если проблему не решать.
5. **Why now.** Почему решать именно сейчас — тренды, технологии, изменения в аудитории.

Опциональные:

- **Scope boundaries.** Чего мы точно НЕ решаем (помогает избежать раздувания).

## Content Rules

- **Проблема сформулирована как явление, не решение.** Плохо: «Нужен сервис для...». Хорошо: «Фрилансеры теряют время на согласование правок».
- **Есть конкретная аудитория.** Общие «все пользователи» — плохо. Должна быть узнаваемая группа.
- **Есть измеримое последствие.** «Теряют время» — ок, но лучше «теряют 3–5 часов в неделю». Оценка грубая допустима.
- **Без технологий.** PS не упоминает конкретные инструменты, стек, API.
- **Длина 200–500 слов.** Длиннее — признак, что намешаны несколько проблем.

## Relationships

**Входящие:** нет (корневой артефакт).

**Исходящие (каскад при изменении PS):**
- → MR (контекст исследования рынка)
- → CA (кого считать конкурентами)
- → SEG-* (целевая аудитория)
- → VP-* (ценностное предложение)
- → HYP-* (предпосылки гипотез)
- → MVP (что именно валидируем)

**Cascade impact при изменении:** все downstream артефакты переходят в `draft` со статусом «требует пересмотра», запускается P6 Pivot Cascade (пока out-of-scope v1, manual).

## Review Level: 🟠 Strategic

Почему не 🔴 Critical: PS не имеет технической зависимости, которая сломается при ошибке. Но почему 🟠: стратегическое решение о направлении продукта требует явного подтверждения человеком. Ассистент формирует draft через диалог, человек approves по факту соответствия реальности.

## Lifecycle States

```
draft ──(G1 approve)──▶ active ──(pivot)──▶ pivoted ──▶ archived
   ▲                       │
   └──(rework)─────────────┘
```

- **draft** — в процессе обсуждения, не используется downstream
- **active** — единственное состояние для работы; остальные артефакты создаются только при active PS
- **pivoted** — устарел после pivot; хранится для истории; создан новый PS
- **archived** — продукт закрыт

## Examples

**Good (сокращённо):**
```yaml
---
id: PS
type: problem-statement
title: "Фрилансеры-переводчики теряют часы на согласование правок с клиентами"
status: active
created: 2026-05-01
updated: 2026-05-01
version: 1
---

## Problem
Фрилансеры-переводчики (~50 000 в русскоязычном сегменте) получают правки
от клиентов через email/Telegram в произвольном формате...

## Context
Проблема проявляется на проектах длительностью > 1 недели, где клиент
возвращается с уточнениями 3-5 раз...

## Current alternatives
1. Google Docs с комментариями — теряется история
2. Trados Studio — для переводчиков в агентствах, не для фрилансеров
...

## Consequences
Переводчик теряет 3-5 часов в неделю на переключения контекста...

## Why now
Рост ИИ-переводчиков оттесняет фрилансеров в нишу "human touch",
где скорость коммуникации с клиентом критична...
```

**Anti-example:**
```
title: "Нужен сервис для управления переводческими проектами"   ❌ решение, не проблема
title: "Всем переводчикам тяжело работать"                        ❌ общая аудитория
title: "Для фрилансеров нет хорошего инструмента"                 ❌ пессимизм без факта
```

## Common Mistakes

1. **Смешение проблем.** Две разные боли → два разных продукта. Решение: разделить PS на две версии продукта или выбрать одну основную.
2. **Продукт-центризм.** «Проблема в том, что нет приложения X». Проблема — у аудитории, не в отсутствии продукта.
3. **Слишком широко.** «Малый бизнес не может расти» — ничего не значит. Нужна конкретная механика.
4. **Слишком узко.** «Васе нужно переводить 3 документа в день» — не продукт, а фриланс-заказ.
5. **Слишком технологично.** Упоминание API, базы, UI — преждевременное ограничение решения.
