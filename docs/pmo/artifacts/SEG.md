# SEG-* — Segment & JTBD

> **Тип:** segment
> **Домен:** D1 — Product Discovery & Strategy
> **Review:** 🟠 Strategic
> **Cardinality:** 2–4 на продукт (рекомендуется ≤3 для MVP, ≤5 для зрелого)
> **Владелец:** Product Module / процесс P1 Discovery, шаг D1.4

## Purpose

Определение функционального **сегмента пользователей** + его **Jobs-to-be-Done (JTBD)**. Заменяет классические User Personas — фокус на **функциональных потребностях**, а не демографических характеристиках. Каждый JTBD порождает потенциальные фичи.

## Frontmatter Schema

```yaml
---
id: SEG-<NNN>                       # SEG-001, SEG-002, ...
type: segment
title: "Короткое имя сегмента"      # Например: "Freelance translators"
status: draft | active | deprecated
priority: primary | secondary | exploratory
value_proposition: VP-<NNN>         # Обязательная ссылка на связанный VP
jtbd_count: N                       # количество JTBD в сегменте
confidence: high | medium | low                  # C2 modification — обязательно
confidence_notes: "string"                       # required если confidence != high
created: YYYY-MM-DD
updated: YYYY-MM-DD
version: 1
---
```

## Body Structure

Обязательные секции:

1. **Profile.** Функциональное описание: роль, контекст работы, ключевые задачи, с которыми сталкиваются.
2. **Size & sources.** Оценка размера сегмента с источниками (из MR или дополнительно).
3. **Jobs-to-be-Done.** 2–6 JTBD в формате: «Когда я [ситуация], я хочу [действие], чтобы [результат]».
4. **Pain points.** 3–5 ключевых болей в работе (обобщение, не жалобы).
5. **Current tooling.** Чем пользуются сейчас для решения этих JTBD.
6. **Validation data.** Наблюдения, интервью, статистика, подтверждающие реальность сегмента.

Опциональные:

- **Behavioral patterns** (как именно работают с инструментами)
- **Willingness to pay** (если известно)
- **Adjacent segments** (кто рядом, но не попадает сюда)

## Content Rules

- **Сегмент — функциональный, не демографический.** Плохо: «Женщины 25–35». Хорошо: «Фрилансеры, ведущие 3+ параллельных проекта».
- **JTBD — через функцию, не демографию.** «Юрист, которому нужно...» — плохо. «Когда я проверяю контракт, я хочу...» — хорошо.
- **Каждый JTBD имеет измеримый результат.** «Чтобы быстрее» — плохо. «Чтобы сократить согласование с 5 раундов до 2» — хорошо.
- **Пересечения сегментов явно отмечены.** Один пользователь может быть в двух сегментах (например: и freelance, и agency-member). Отмечается в `adjacent_segments`.
- **Нет product-specific предположений.** SEG описывает аудиторию, не наш продукт.

### JTBD-формат

Каждый JTBD структурирован:

```markdown
### JTBD-1: Сбор правок от нескольких клиентов
- **Ситуация:** работаю с 3–5 клиентами параллельно, каждый шлёт правки своим каналом
- **Желание:** централизовать все правки в одном месте с привязкой к контексту документа
- **Результат:** не теряется ни одна правка, время обработки < 15 минут на батч
- **Фрустрация сейчас:** переключение между Telegram, email, Google Docs; теряется 2–3 часа в неделю
```

## Relationships

**Входящие:**
- ← PS (целевая аудитория продукта)
- ← MR (размер, характеристики)
- ← CA (кого конкуренты обслуживают)

**Исходящие:**
- → VP-{NNN} (каждый сегмент имеет своё ценностное предложение; 1:1)
- → HYP-* (гипотезы про поведение сегмента)
- → FM-* (feature связаны с конкретными JTBD)
- → SC-* (сценарии отражают поведение ролей из сегмента)

**Cascade impact при изменении:**
- Добавление JTBD → проверить, покрыт ли существующими FM
- Удаление сегмента → все FM для него в `requires_review`
- Смена priority → пересмотр MVP scope

## Review Level: 🟠 Strategic

Ошибочный сегмент каскадно искажает всё downstream. Человек подтверждает на основе **реальных наблюдений**, не гипотез.

## Lifecycle States

```
draft ──(G4 approve)──▶ active ──▶ deprecated (если сегмент отмерся)
                          │
                          └──(priority change)──▶ active v2
```

### Priority levels

- **primary** — основной сегмент MVP; FM для него — MUST have
- **secondary** — дополнительный; FM для него — SHOULD have
- **exploratory** — проверяем гипотезу о соответствии; FM — COULD have

## Examples

**Good (фрагмент):**
```yaml
---
id: SEG-001
title: "Freelance translators (рус-англ)"
status: active
priority: primary
value_proposition: VP-001
jtbd_count: 4
---

## Profile
Фрилансеры-переводчики, работающие с 3–8 клиентами параллельно,
специализация RU↔EN + 1 дополнительный язык. Работают в основном
из дома, через биржи (Upwork, Fl.ru) и прямые контакты.
Средний опыт: 3–5 лет. Возраст 25–45, но это вторично.

## Size & sources
~12 000 активных (>= 10 проектов/год) в русскоязычном сегменте.
Источники: MR § «Growth segments» + данные Upwork freelancer database.

## JTBD-1: Сбор правок от нескольких клиентов
- Ситуация: 3–5 клиентов, каждый шлёт правки своим каналом
- Желание: централизованно с контекстом
- Результат: < 15 мин на батч правок
- Фрустрация: теряет 2–3 часа/неделю на переключения

## JTBD-2: Согласование терминов с клиентом
...

## JTBD-3: Отчётность перед клиентом
...

## JTBD-4: Контроль сроков
...

## Pain points
1. Потеря контекста правок при переключении каналов
2. Клиент не принимает работу из-за несогласованных терминов
...

## Current tooling
Google Docs (basic), Trados (сложно для freelance), ChatGPT (для проверок)

## Validation data
5 интервью с фрилансерами через Telegram-чат (2026-04-10..15).
Все 5 упомянули JTBD-1 как главный pain.
```

**Anti-example:**
```
title: "Все пользователи"                  ❌ не сегмент
title: "Миллениалы"                        ❌ демографически
title: "Freelancers"                       ❌ слишком широко
JTBD: "Хочет работать эффективнее"         ❌ без ситуации и результата
```

## Common Mistakes

1. **Слишком много сегментов.** 6+ сегментов для MVP = не сфокусированы. Начни с 1-2.
2. **Демография вместо функции.** Возраст, пол, доход — не определяют сегмент.
3. **JTBD = фичи.** JTBD описывает нужду, фича — решение. Не смешивать.
4. **Сегмент без валидации.** Сегмент «на бумаге» без интервью/данных — гипотеза, не факт.
5. **Дублирование с VP.** VP принадлежит сегменту, не наоборот. Избегать переклинивания.
