# VP-* — Value Proposition

> **Тип:** value-proposition
> **Домен:** D1 — Product Discovery & Strategy
> **Review:** 🟠 Strategic
> **Cardinality:** 1:1 c SEG-*, т.е. 1 VP на сегмент
> **Владелец:** Product Module / процесс P1 Discovery, шаг D1.4a

## Purpose

**Мост между стратегией и фичами**. Формулирует, какую **ценность** продукт создаёт для конкретного сегмента: выполняет JTBD лучше существующих альтернатив + избавляет от pain points. VP определяет, **за что платит или почему выбирает** наш продукт пользователь данного сегмента.

## Frontmatter Schema

```yaml
---
id: VP-<NNN>                        # VP-001, VP-002, ...
type: value-proposition
title: "Короткая формулировка VP"
segment: SEG-<NNN>                   # обязательная 1:1 связь
status: draft | active | deprecated
confidence: high | medium | low                  # C2 modification — обязательно
confidence_notes: "string"                       # required если confidence != high
created: YYYY-MM-DD
updated: YYYY-MM-DD
version: 1
---
```

## Body Structure

Обязательные секции:

1. **Statement.** Одно предложение формата: «Для [сегмента], которые [JTBD], наш продукт — [категория], который [key benefit], в отличие от [альтернатив], мы [differentiator]».
2. **Jobs addressed.** Какие JTBD из SEG-* решает этот VP (может быть не все — явно отмечать).
3. **Pain relievers.** Что из pain points SEG-* устраняется и как.
4. **Gain creators.** Какие новые возможности / улучшения даёт помимо устранения болей.
5. **Differentiation.** Чем отличается от конкурентов из CA (конкретно).
6. **Evidence.** На чём основана уверенность, что этот VP будет работать (данные из MR, CA, HYP, интервью).

Опциональные:

- **Secondary VPs** (если для разных use case внутри сегмента)
- **Anti-VP** (что мы сознательно НЕ делаем для этого сегмента)

## Content Rules

- **Начинается с сегмента, не с продукта.** «Для фрилансеров, которые...» — хорошо. «Наш продукт — это...» — плохо.
- **Конкретный differentiator.** «Лучше, быстрее, дешевле» — не differentiator. «Workflow с клиентом встроен в инструмент перевода, в отличие от Smartcat, где они разделены» — differentiator.
- **Привязан к реальным JTBD из SEG.** Нельзя изобретать новые JTBD в VP; только то, что уже есть в SEG-*.
- **Не все JTBD обязаны быть покрыты.** Если VP покрывает 2 из 4 JTBD — это нормально. Явно отмечаем какие не покрыты и почему.
- **Не упоминает конкретные фичи.** VP — это обещание ценности, не список функций.

### Formula Value Proposition Canvas

Используем упрощённый Strategyzer canvas:

```
Customer Jobs (из SEG) ←→ Pain Relievers (из VP)
                       ←→ Gain Creators (из VP)
```

## Relationships

**Входящие:**
- ← SEG-{NNN} (один VP на сегмент)
- ← PS (общий контекст проблемы)
- ← CA (основание для differentiation)

**Исходящие:**
- → HYP-* (гипотезы проверяют VP)
- → FM-* (фичи реализуют обещания VP)
- → MVP (определяет, какой VP валидируем в первую очередь)

**Cascade impact при изменении:**
- Изменение VP → проверить все FM, которые его реализуют (MUST have фичи должны всё ещё выполнять VP)
- Инвалидация HYP, проверявшей VP → флаг на пересмотр VP

## Review Level: 🟠 Strategic

Человек проверяет, что VP отражает **реальное** конкурентное преимущество (не декларируемое). Ассистент собирает материал, но окончательная формулировка — стратегическая.

## Lifecycle States

```
draft ──(G4a approve)──▶ active ──(SEG deprecated)──▶ deprecated
                          │
                          └──(competitor catches up)──▶ draft (pivot VP)
```

## Examples

**Good (фрагмент):**
```yaml
---
id: VP-001
title: "Фрилансер-переводчик не теряет правки от клиентов"
segment: SEG-001
status: active
---

## Statement
Для **фрилансеров-переводчиков (SEG-001)**, которые **теряют время на сбор 
и согласование правок от 3+ клиентов параллельно**, наш продукт —
**workflow-инструмент, встроенный в процесс перевода**, который **централизует
правки с сохранением контекста документа**, в отличие от **Smartcat и Trados,
ориентированных на agency-flow**, мы **оптимизируем именно solo-фрилансерский
случай с множественными клиентами**.

## Jobs addressed
- JTBD-1 (сбор правок) — основной
- JTBD-2 (согласование терминов) — частично
- JTBD-3 (отчётность) — не покрывается явно (опционально, через экспорт)
- JTBD-4 (контроль сроков) — не покрывается (отложено до v2)

## Pain relievers
- Устраняет переключение между Telegram/email/Google Docs (pain #1)
- Устраняет потерю контекста при согласовании (pain #2)

## Gain creators
- Видимость всех активных проектов в одном дашборде
- Автоматическая история правок для отчётности

## Differentiation
- vs Smartcat: мы оптимизируем для 1 переводчика × много клиентов; Smartcat — 
  для agency × много переводчиков
- vs Trados: мы web-first; Trados требует desktop install
- vs Google Docs: мы встраиваем в процесс перевода (TM, glossary), 
  не универсальный текст

## Evidence
- MR показывает смещение freelancers в нишу "human touch" из-за ИИ
- CA показывает gap: никто не фокусируется на freelance × multiple clients
- HYP-001 проверяет конверсию freelance → paid на 10% за 3 месяца
```

**Anti-example:**
```
"Наш продукт лучший для всех переводчиков"              ❌ нет сегмента
"Мы используем AI для автоматизации"                     ❌ фича, не ценность
"Быстрее и удобнее конкурентов"                          ❌ без differentiation
"Решаем все проблемы перевода"                           ❌ слишком широко
```

## Common Mistakes

1. **VP = список фич.** VP описывает результат для пользователя, не механику.
2. **VP без привязки к сегменту.** Один VP на всех пользователей = неработающий VP.
3. **Ложное differentiation.** «Мы используем AI» — все используют, не отличает.
4. **VP без evidence.** Если нет данных, это не VP, а hypothesis — создавай HYP.
5. **Слишком много JTBD покрыто.** VP, покрывающий 10 JTBD — обычно плохо, нужна специализация.
