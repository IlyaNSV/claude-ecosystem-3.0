# Domain Expertise — Ecosystem 3.0

> **Версия:** 1.0 (2026-07-10)
> **Назначение:** единый справочник концепта «доменная экспертиза» — в каких доменах
> ИТ-продуктов экосистема сильна, как идея классифицируется по домену и как работает
> гейт соответствия (Domain Fit) на входе в Discovery.
> **Введён:** DEC-DEV-0169.
> **Принцип:** в отличие от `product_class` (descriptive, никогда не gate'ит), Domain Fit —
> **единственный класс-гейт экосистемы**: ниже порога поток Discovery останавливается до
> явного решения владельца. Гейт информирует и требует решения — но финальное слово
> всегда за владельцем (override легален и фиксируется).

---

## 1. Зачем

Оценка универсальности 2026-07-10 (96 подкатегорий × 10 критериев,
`dev/universality-assessment/REPORT.md`) показала водораздел: продукты с
**поведенческим ядром** (workflow / бизнес-правила / жизненные циклы) ложатся на
экосистему на 70–92 балла, продукты с **алгоритмическим ядром** (математика / real-time /
геймплей / системное ПО) — на 30–45 независимо от категории. До DEC-DEV-0169 вход идеи
(`/product:init`) никак это не проверял: можно было завести полный Discovery-цикл для
продукта, чью суть экосистема выразить не может, и обнаружить это лишь на handoff.

Domain Fit закрывает зазор **промежуточным этапом анализа**: идея классифицируется до
подкатегории (максимальная детализация — 96 позиций), берётся её совокупный балл из
реестра, и при балле ниже порога поток останавливается с показом ограничителей и
вариантами решения.

## 2. Где живёт (SSOT)

- **Данные:** [`domain-expertise-registry.md`](domain-expertise-registry.md) — матрица
  96 подкатегорий (категории A–P) × критерии C1–C10 + агрегат + ограничитель/сила.
- **Вердикт по проекту:** блок `domain_fit` в **`.claude/product.yaml`** (проектный
  конфиг, как `product_class`) — доменная принадлежность стабильный проектный факт,
  не версионируемый артефакт и не поле PS (PS обязан быть tech-free).
- **Захват:** шаг **D1.0b Domain Fit Assessment** в Discovery (`/product:init`), сразу
  после D1.0 Product Classification (архетип сужает поиск подкатегории) и до
  D1.1 Problem Discovery.
- **Методология диалога:** `skills/product/domain-fit.md` (режимы `discovery` и `assess`).
- **Вывод (derive, не переписывать):** зафиксированные `limiters` — вход для суждения при
  MVP-скоупинге и NFR-review; значение нигде не дублируется, только читается из
  `product.yaml`.

### Схема блока (канон — НЕ варьировать имена полей)

```yaml
domain_fit:
  subcategory: A1                  # ID из реестра (A1..P4) | `unmapped`
  subcategory_label: "B2B SaaS (vertical/horizontal)"
  score: 92                        # агрегат подкатегории из реестра (0-100)
  threshold: 75                    # действующий порог гейта (дефолт 75)
  verdict: fit                     # fit | conditional-fit | misfit | unmapped
  decision: proceed                # proceed | proceed-with-risks | adapted | aborted
  limiters: ""                     # главный ограничитель из реестра + гибрид-примечания
  hybrid_components: []            # (опц.) [{subcategory, score, role}] для гибридов
  confidence: high                 # high | medium | low — уверенность классификации
  source: discovery                # discovery | manual
  registry_version: "2026-07-10"   # версия реестра на момент оценки
  assessed_at: "2026-07-10"
  notes: ""                        # ОБЯЗАТЕЛЕН если subcategory=unmapped или decision != proceed
```

Блок целиком **опционален**: отсутствие = поведение экосистемы 1:1 как до 0169
(soft-миграция по прецеденту `product_class`, DEC-DEV-0079). Backfill для существующих
проектов — режим `assess` skill'а.

## 3. Модель оценки

### 3.1 Классификация — по ядру, не по обвязке

Подкатегория выбирается по тому, **где лежит ядро ценности продукта**, а не по его
обвязке. Почти у любого продукта есть выразимая половина (дашборд, CRUD-настройки,
биллинг) — она не делает продукт «своим» для экосистемы, если суть в алгоритмическом
ядре. Пример из реестра: игровой backend (D6 = 74) ≠ сама игра (D2 = 39); IoT-платформа
(J2 = 68) расщепляется на облачную половину (sweet spot) и firmware (провал).

### 3.2 Гранулярность — подкатегория, всегда

Оценка ведётся по подкатегории (96 позиций), не по категории (16 позиций): разброс внутри
категории достигает 40+ баллов (пример: E1 CLI-утилита = 84 против E5 билд-система = 54 в
одной категории DevTools). Средний балл категории в реестре — контекст, не основание
вердикта.

### 3.3 Гибриды — декомпозиция

Если идея расщепляется на компоненты разных подкатегорий (вывод 3 исходной оценки),
skill фиксирует компоненты в `hybrid_components` с ролями (`core` — ядро ценности /
`supporting` — обвязка). **Вердикт считается по компоненту `core`.** Если равноправных
ядер два — вердикт по слабейшему из них (консервативно), оба в `hybrid_components`,
ограничители обоих в `limiters`.

### 3.4 Незнакомый домен — `unmapped`

Реестр конечен, мир — нет. Идея, не ложащаяся ни в одну подкатегорию, получает
`subcategory: unmapped` + обязательный `notes`. Гейт при этом **деградирует в advisory**
(блокировать по отсутствию данных нельзя — открытый мир, тот же принцип, что
`archetype: other`): skill показывает ближайшие подкатегории-якоря и их баллы как
ориентир, решение фиксируется, поток продолжается по выбору пользователя.

## 4. Гейт-семантика

Порог: **75** (дефолт; override — поле `threshold` в блоке, меняется только владельцем
осознанно, с `notes`). Шкала реестра: ≥90 sweet spot · 70–89 хорошо · 50–69 частично ·
30–49 слабо · <30 чужая парадигма.

| Балл | verdict | Поведение гейта |
|---|---|---|
| ≥ threshold | `fit` | Пройден. Одна строка подтверждения (подкатегория + балл + сила из реестра), Discovery продолжается без пауз. |
| 50 … threshold−1 | `conditional-fit` | **Стоп.** Показать разбивку по критериям + ограничитель из реестра; предложить решения (см. ниже). |
| < 50 | `misfit` | **Стоп + прямое предупреждение:** ядро продукта — вне парадигмы экосистемы; перечислить, ЧТО именно она не выразит. Решения те же, но рекомендация по умолчанию — `adapted` или `aborted`. |
| нет данных | `unmapped` | Advisory (см. §3.4), не блокирует. |

### Решения на гейте (фиксируются в `decision`)

1. **`adapted`** — перекроить идею под выразимую половину (гибрид-паттерн: облачная
   половина IoT вместо firmware, game-backend вместо игры, продукт-обёртка вместо
   ядра-алгоритма). После адаптации — повторная классификация, новый вердикт.
2. **`proceed-with-risks`** — осознанно продолжить: ограничители из реестра копируются в
   `limiters`, `notes` обязателен. Экосистема ведёт выразимую часть, невыразимое ядро —
   явно за скобками handoff (владелец принял это письменно).
3. **`aborted`** — остановиться; Discovery не продолжается, блок остаётся в
   `product.yaml` как след решения.

Прошедший гейт фиксирует `decision: proceed`.

### Анти-принципы гейта

- Гейт **не оценивает качество идеи** — только соответствие домена сильным зонам
  инструмента. Отличная идея в слабом домене — повод взять другой инструмент, не повод
  хоронить идею.
- Гейт **не пересчитывает баллы** — только lookup из реестра. Сомнение в балле →
  протокол обновления (§6), не подкрутка на месте.
- Override владельца **всегда легален** — гейт производит информированное решение,
  а не запрет.

## 5. Отличие от product_class

| | `product_class` (0079) | `domain_fit` (0169) |
|---|---|---|
| Вопрос | какой **формы** продукт (архетип/фасеты) | в каком **домене** продукт и силён ли тут инструмент |
| Словарь | открытый, любое значение легально | закрытый реестр 96 позиций + `unmapped`-выход |
| Характер | descriptive, только сеет дефолты | **гейт с порогом** (75) + owner-override |
| Шаг | D1.0 | D1.0b (использует archetype из D1.0 как подсказку) |
| Потребители | NFR/тест-дефолты, handoff hint | стоп/го-решение на входе + `limiters` для скоупинга |

Оба живут в `product.yaml`, оба опциональны (soft-миграция), оба пишутся своим skill'ом.

## 6. Протокол обновления реестра

Баллы — снапшот способностей экосистемы на дату версии реестра. Они **устаревают в
хорошую сторону**: закрытие пробелов (contract/schema-спеки, eval-спеки, non-market
discovery, перф-бенчмарк NFR, compliance-матрица — вывод 7 исходной оценки) поднимает
целые классы подкатегорий.

1. **Точечная правка баллов запрещена** — только полная или секционная переоценка тем же
   методом (независимые оценщики + детерминированная агрегация; метод описан в
   `dev/universality-assessment/REPORT.md`).
2. Переоценка → новая дата в шапке реестра + запись DEV_JOURNAL со ссылкой на прогон.
3. `registry_version` в блоках `domain_fit` существующих проектов при этом НЕ трогается —
   он честно фиксирует, по какой версии принималось решение. Пересмотр вердикта проекта —
   по желанию владельца (`assess` режим skill'а поверх новой версии).
4. **Добавление подкатегории** (новый домен в реестр): оценить по всем C1–C10 тем же
   методом, вставить строку в соответствующую категорию, bump версии. Категорийные
   средние в заголовках секций пересчитать.

## 7. Как потребляется

- **D1.0b Discovery** (`skills/product/domain-fit.md`, режим `discovery`) — классификация
  идеи → подтверждение пользователем → lookup → вердикт/гейт → запись в `product.yaml`.
- **Backfill / переоценка** — режим `assess` того же skill'а (прямая просьба пользователя;
  проект с уже пройденным Discovery; переоценка после bump'а версии реестра или пивота).
- **MVP-скоупинг / NFR-review** — `limiters` из блока служат явным напоминанием, какие
  части продукта экосистема ведёт, а какие за скобками (advisory, читается по месту).
- **Handoff** (DEC-DEV-0172) — `skills/product/handoff-generator.md` читает блок и, **только
  если `limiters` непуст**, эмитит строку `Domain fit:` в §1 + frontmatter-блок `domain_fit`
  + bullet в §12 (receiver видит, что осознанно за скобками поведенческого контура). Чистый
  fit не эмитится — нет downstream-сигнала. См. `handoff-spec.md §1/§5/§12`.
